---

TLになった初日、最初にやったのはgit logを見ることでした。

コミット履歴はある。コードはある。でも、何かがおかしい。ファイル一覧を眺めると、それが何かわかりました。`tests/`ディレクトリがない。`.github/workflows/`もない。`.eslintrc`も`.flake8`も何もない。そして`.env`がリポジトリのルートに堂々とコミットされていました。

前任のTLに「引き継ぎの時間はとれますか」と聞いたら、「動いてるから大丈夫です」と言って退職しました。

あなたがこの記事を読んでいるなら、おそらく似た状況にいると思います。テストゼロ。CIなし。コードスタイルはバラバラ。上からは「品質を上げろ」と言われているけれど、何から手をつければいいかわからない。

この記事はそういうTLのために書きました。

---

## なぜ品質基盤のないチームが生まれるのか

責めたいわけではありません。でも構造を理解しないと、対策が打てません。

### 「後で入れる」が永遠に来ない

品質基盤のないチームに「なぜlintを入れなかったのか」と聞くと、たいてい「最初は小さかったので」という答えが返ってきます。

立ち上げ期は機能開発が最優先です。lintやCIは「いつでも後で入れられる」という認識で、後回しになります。でも後回しにした瞬間、それは「負債」になります。負債は利子を生みます。コードが増えるほど、後から入れるコストが上がります。技術的負債の利子は複利です。

結果として「後で入れる」タイミングは永遠に来ません。

### 個人の規律に頼る品質管理の限界

lintもCIもテストもない状態での品質管理は、「メンバーが気をつける」ことに依存します。

これは機能しません。

人間は疲れます。締め切り前は焦ります。「今回だけ」と思ってprint文を残します。例外を握りつぶします。そういう判断を毎回正しくできる人間はいません。

「品質は仕組みで担保する」という考え方に転換する必要があります。個人の努力ではなく、悪いコードが入りにくい構造を作ることが、TLの仕事です。

### データで見る: 78件のバグのうち46件は仕組みで防げた

ある業務システムをAI支援で開発したとき、78件のバグを記録し、根本原因で分析した結果、**59%（46件）は品質基盤（lint・CI・テスト）の導入だけで防止可能でした**。詳細な原因分析はArt.7「78バグから導いた7つの設計原則」で解説しています。

「品質を上げろ」という指示に、最も直接的に答える方法は、仕組みを作ることです。

## Day 0の戦略: 4時間で基盤を作る

「品質基盤の整備」というと、数週間かかる大プロジェクトに聞こえます。でもそうではありません。最低限の仕組みを動かすだけなら、4時間でできます。

重要な原則が2つあります。

**1. 既存コードには触らない**

レガシーコードをすべてlintに通そうとすると、膨大なエラーが出ます。それを全部直してからlintを有効化しようとすると、何週間もかかります。そしてその間にlintは使われません。

解決策は「新規ファイルだけstrictにする」戦略です。既存コードはとりあえず除外して、今後追加するファイルだけにルールを適用します。レガシーコードには段階的に対応していけばよい。

**2. 優先順位を間違えない**

lint → CI → テスト基盤 → ログ基盤の順番で入れます。理由は後述しますが、この順番には根拠があります。

## Step 1: lint（2時間）― 78件のうち35件を自動検出する

まずlintを入れます。これが最優先の理由は、設定ファイルを1つ追加するだけで即座に効果が出るからです。CIやテストは動かすまでに時間がかかりますが、lintはエディタと連携すれば書いた瞬間にフィードバックが返ります。

各言語で有効にすべきルールの要点は以下の通りです。

- **Python (ruff)**: `T20`（print残留検出）、`B`（bare except検出）、`F`（未使用変数・import）。既存コードは `exclude` で除外し、新規ファイルだけにまず適用する。
- **TypeScript (ESLint + tsconfig)**: `no-console`、`no-explicit-any`、tsconfig に `"strict": true`。`ignorePatterns` でレガシーディレクトリを除外。
- **Go (staticcheck / golangci-lint)**: `go vet`、`errcheck`（エラー未処理の検出）、`--skip-dirs legacy` で既存コードを回避。

これだけで、print残留(T20)とbare except(B001〜B029)がエラーとして検出されます。78件のバグのうち35件はこれで防げました。

コピペ可能な完全な設定ファイル（pyproject.toml / .eslintrc.json / .golangci.yml）は、バグ分析の元記事であるArt.1に公開しています。既存コードを段階的にlintに通す方法も含めて解説しているので、設定を一から書く場合はそちらを参照してください。

→ **Art.1「78バグの全分類」**（Qiita、無料）: 設定ファイルのコピペ元はここ

## Step 2: CI（1時間）― lintを「自動で走らせる」

lintを入れただけでは不十分です。ローカルで動かすだけなら、忘れる人が必ず出ます。CIに組み込んで、プルリクエストのたびに自動実行する仕組みを作ります。

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [main, develop]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install ruff
      - run: ruff check .

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -r requirements.txt
      - run: pytest tests/
```

TypeScriptなら`npm run lint && npm run type-check`、Goなら`go vet ./... && go test ./...`に置き換えるだけです。

このCIが動けば、「マージしたら壊れる」問題の多くが解決します。プルリクエストを出した時点でlintが走り、エラーがあればマージできません。CIが入ったその日から、print文とbare exceptは本番に入らなくなります。

## Step 3: テスト基盤（1時間）― 「動く場所」を作る

テストゼロの状態で「テストを書いて」と言っても書かれません。理由は単純で、テストを実行する環境がないからです。テストファイルを作ってもどうやって動かせばいいかわからない。

まず「テストが動く環境」を作ることが先決です。テストの中身はあとでいい。

各言語の最小構成です。「環境が動く」ことを確認するプレースホルダーテストを1件書くところから始めます。

**Python (pytest):**

```python
# tests/test_placeholder.py
def test_env_is_working():
    """最初のテスト。環境が動くことを確認する。"""
    assert 1 + 1 == 2
```

```ini
# pytest.ini
[pytest]
testpaths = tests
addopts = -v --tb=short
```

**TypeScript (vitest):**

```typescript
// src/placeholder.test.ts
import { describe, it, expect } from "vitest";

describe("env", () => {
  it("is working", () => {
    expect(1 + 1).toBe(2);
  });
});
```

**Go (標準testing):**

```go
// env_test.go
package main_test

import "testing"

func TestEnvIsWorking(t *testing.T) {
    if 1+1 != 2 { t.Fatal("環境が動いていません") }
}
```

中身はあとでいい。「テストが動く場所」を作ることが目的です。CIのテストジョブはStep 2のワークフローにすでに含まれています。

## チーム導入の現実: 「テスト書いて」では動かない

ここまでの設定をひとりで作ることはできます。問題は、チームに定着させることです。

### 「テスト書いて」ではなく「CIが落ちたら直して」

「これからテストを書くようにしてください」という指示は機能しません。なぜなら、テストを書かなくてもコードは動くからです。

CIが落ちたら直さなければいけない。これは違います。CIが落ちている状態でマージはできません。**テストを書かないことのコスト**が、初めて可視化されます。

段階的にCIの要件を上げていきます。最初はlintだけ。次にテストが0件でなければOK。その次にカバレッジ50%以上。このように、徐々に基準を上げていくと、メンバーは少しずつテストを書くようになります。

### pre-commitで自動化する

「コミット前にlintを確認する」という行動を、人の習慣に依存させません。

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: detect-private-key  # .envのコミット防止
```

```bash
# チームメンバーの環境にインストール
pip install pre-commit
pre-commit install
```

これでコミットのたびにlintが自動で走ります。lintエラーがあればコミットできません。「確認するのを忘れた」が起きなくなります。

### 最初の成功体験を作る

Week 1の目標は、「CIが1件のバグを止める」という成功体験を作ることです。

CIを入れた後、最初にプルリクエストが上がったとき、lintが何かを検出したとします。その瞬間、チームに言います。「CIが止めてくれましたね。本番に行く前に見つかってよかった」と。

この体験が、CIへの信頼を作ります。「CIは邪魔なもの」ではなく「CIは守ってくれるもの」という認識が、少しずつ広がります。

## 4週間ロードマップ

焦らずやります。4週間で基盤を固めることが目標です。

| 週 | テーマ | やること | 確認基準 | やらないこと |
|----|--------|----------|----------|------------|
| Week 1 | lint + CI（4時間） | ruff/ESLint/staticcheckの設定、GitHub Actions最小構成、pre-commit設定 | CIが1回動いた、lintエラーが1件検出された | 既存コードのlint修正、テストの要求 |
| Week 2 | テスト基盤 + 最初の10テスト | pytest/vitest/go testの設定、CIにテスト追加、自分で10件書く（手本） | `pytest tests/`が動く、10件のテストが存在する | カバレッジ目標設定、既存コードへのテスト強制 |
| Week 3 | pre-commit + レビュー習慣 | pre-commitを全員に展開、PRテンプレートに「テスト追加したか？」を追加 | 全員のローカルでpre-commitが動く、新規PRの50%以上にテストファイル | テストなしPRのreject（まだ早い） |
| Week 4 | カバレッジ閾値 + 効果測定 | CIにカバレッジ計測を追加（閾値30%）、「CIが止めたバグ件数」を集計・共有 | カバレッジが計測できている、チームが「CIが役に立つ」と感じている | 閾値を一気に上げる |

## まとめ: 今日の4時間でできること

TLになったその日にできることがあります。

1. **ruff / ESLint / staticcheck** を入れて、lintを動かす（1〜2時間）
2. **GitHub Actions** でlintをCIに組み込む（1時間）
3. **テスト環境** を作って、プレースホルダーテストを1つ書く（1時間）

これだけで、「マージしたら壊れる」のリスクが大幅に下がります。print残留とbare exceptは自動で検出されます。テストを書く「場所」ができます。

「品質を上げろ」という指示に、最初の答えを出せます。

4週間後、チームは変わっています。少なくとも、前任のTLが残していったような状態には戻りません。

一緒にやりましょう。
