## TL;DR

- AIは「書かれていないこと」を推測で補完する。その推測があなたの意図と一致する保証はない
- Example Mapping（BDD由来）は **Story → Rule → Example → Question** の4要素で仕様を構造化する手法
- 具体例（Example）を書く過程で、仕様の「穴」が自然に見つかる。開発前に潰せる
- Webアプリ・REST API・CLIツール、どの領域でも同じ構造で使える
- この記事のExample Mapテンプレートはそのままコピペしてプロジェクトに導入可能

## この記事でできること

| やりたいこと | この記事で得られるもの |
|---|---|
| AIへの仕様指示の精度を上げたい | Example Mapping の実践手順（5ステップ） |
| 仕様の抜け漏れを事前に防ぎたい | 4種類のカードによる構造的な洗い出し方法 |
| コピペで使えるテンプレートが欲しい | 3領域（Web・API・CLI）のExample Map実例 |
| Before/Afterで効果を確認したい | 一行指示 vs Example Map付き指示の比較 |

「お問い合わせ一覧を作って」

この一言から何が生まれるか知っていますか？ 15件の仕様バグです。

私がAIと一緒に開発した業務システムで、リリース後に78件のバグを分類しました。そのうち15件が「仕様の認識齟齬」でした。コードが間違っていたわけではありません。AIは私が書いた通りに実装しました。問題は、私が書いた通りの仕様が、私の頭の中にあった仕様と違っていたことです。

この記事では、その15件の齟齬がなぜ起きたのかを分析し、Matt Wynne（Cucumber）が提唱したExample Mappingという手法を使って「仕様の抜け漏れ」を構造的に潰す方法を紹介します。Webアプリ、REST API、CLIツールの3つの実演を通じて、言語やフレームワークに依存しない「仕様の書き方」を身につけていただければと思います。

---

## 15件の仕様齟齬の実態

まず、15件の中から代表的なものを4つ紹介します。どれも「あるある」だと思います。

### 齟齬1: ソート順が決まっていなかった

私の指示は「お問い合わせ一覧を表示してください」でした。

AIが実装したのは、データベースから取得した順番でそのまま表示するコードでした。つまり、INSERTされた順番です。私が頭の中で想像していたのは「新しい順（作成日時の降順）」でしたが、それを一言も書いていませんでした。

ユーザーが画面を開いたとき、3ヶ月前の古いお問い合わせが一番上に表示されていて、「これ、壊れてますか？」と聞かれました。壊れていません。仕様が壊れていたのです。

### 齟齬2: ページネーションの仕様が曖昧だった

「一覧にページネーションをつけてください」と伝えました。

AIは1ページ10件、URLパラメータで`?page=2`に切り替える実装をしました。私が想定していたのは1ページ20件で、ボタンクリックでの画面遷移でした。どちらも「ページネーション」です。どちらが正解かはプロジェクトの要件次第ですが、その要件を私は書いていませんでした。

さらに厄介だったのは、ページネーションの「境界条件」です。

- 0件のとき、ページネーションのUIは表示されるのか？
- ちょうど10件のとき、「次のページ」ボタンは出るのか？
- 総件数は表示するのか？

これらは全て、「ページネーションをつけて」という一言からは読み取れません。

### 齟齬3: 0件表示が未定義だった

データが1件もないとき、画面に何を表示するかを指定していませんでした。

AIが実装したのは、空のテーブルがそのまま表示されるコードでした。ヘッダー行だけが表示されて、データ行がない状態です。見た目としては「読み込み中なのか？エラーなのか？本当に0件なのか？」が判断できません。

私が期待していたのは「お問い合わせはまだありません」というメッセージの表示でしたが、それを伝えていない以上、AIがそれを実装する理由がありません。

### 齟齬4: 権限による表示制御が抜けていた

「お問い合わせ一覧」と言ったとき、私の頭の中では暗黙的に「管理者向け」のイメージがありました。でも、AIにとっては「誰が見る画面なのか」が未定義のまま実装が進みました。

結果として、一般ユーザーがアクセスしても全てのお問い合わせが表示される実装になっていました。管理者は全件を見られるが、一般ユーザーは自分が投稿したものだけ、という権限制御は仕様に書かれていなかったからです。

本番環境で「他の人のお問い合わせも見えてしまっている」という報告が来たとき、冷や汗が出ました。これは仕様バグであると同時に、セキュリティの問題でもありました。

## なぜAIに意図が伝わらないのか

15件の齟齬を並べてみると、ある共通点が見えてきます。どれも「私が書かなかったこと」が原因です。AIのコードが間違っていたケースは1件もありませんでした。

ここには3つの構造的な原因があります。

### 原因1: AIは「書かれたこと」だけを実装する

人間のエンジニアに「お問い合わせ一覧を作って」と言えば、過去の経験から「ソートは新しい順ですよね？」「0件のときはどうしますか？」と聞き返してくれることがあります。経験豊富なエンジニアほど、この「補完力」が高いです。

AIも補完はします。ただし、その補完はプロジェクト固有の文脈を持っていません。「ページネーションは10件にしよう」という判断はAIの一般的な知識から来ており、あなたのプロジェクトの要件から来ているわけではありません。

書かれていないことを、AIは「わからないから聞こう」ではなく「一般的にこうだろう」という方向で補完します。この補完が私の意図と一致しなかった場合に「仕様齟齬」が生まれます。

### 原因2: 人間同士なら暗黙の了解で補完できる

チーム開発では、長い時間をかけて「暗黙の了解」が蓄積されていきます。

- 「うちのプロジェクトでは一覧は20件表示がデフォルト」
- 「0件のときはempty stateコンポーネントを使う」
- 「管理画面は管理者のみアクセスできる前提」

こうした了解は、仕様書に書かれていなくても、チームの中では共有されています。新しいメンバーが入ってきたときに「あ、それはこうだから」と口頭で伝えるような知識です。

AIにはこの蓄積がありません。毎回の会話がゼロからのスタートです。前回の会話で「うちは20件表示だよ」と伝えても、新しい会話では忘れています。CLAUDE.mdやプロジェクト設定である程度は引き継げますが、「暗黙の了解」の全てを明文化するのは現実的ではありません。

### 原因3: これは言語の問題ではなく、コミュニケーションの構造の問題

この問題は「日本語の書き方が悪い」とか「もっと詳しく書けばいい」というレベルの話ではありません。

根本的な問題は、仕様を伝えるための**構造がない**ことです。

自然言語で「お問い合わせ一覧を作って。ソートは新しい順で、1ページ20件で、0件のときはメッセージを出して、管理者のみアクセスできるようにして...」と書けば伝わります。でも、こうやって思いつくままに羅列すると、必ず漏れが出ます。ソートのことを書いたのに、フィルタリングのことを忘れる。権限のことを書いたのに、権限エラー時の挙動を書き忘れる。

必要なのは、「何を書くべきか」を体系的に洗い出すフレームワークです。

## Example Mappingとは

Example Mappingは、BDD（振る舞い駆動開発）の文脈で生まれた要件整理の手法です。Matt Wynne（Cucumberの共同創設者）が提唱しました。BDDの「Discovery（発見）→ Formulation（記述）→ Automation（自動化）」という3ステップのうち、最初のDiscovery段階で使う手法です。

特徴は、4種類のカードを使って仕様を構造的に整理するところにあります。

| カード | 色 | 役割 | 例 |
|---|---|---|---|
| **Story** | 黄色 | ユーザーストーリー。「誰が」「何をしたいか」を定義する。1つのExample Mapに1つだけ。 | 「管理者として、お問い合わせ一覧を見たい」 |
| **Rule** | 青色 | ビジネスルール。Storyを成立させるための条件や制約。1つのStoryに複数ぶら下がる。 | 「一覧はページネーションされる」「管理者のみアクセスできる」 |
| **Example** | 緑色 | 具体例。各Ruleを具体的なシナリオで表現する。「このインプットのとき、こう動く」形式。 | 「21件以上あるとき → ページネーションが表示される」 |
| **Question** | 赤色 | 未解決の疑問。Exampleを書く過程で浮かび上がった「まだ決まっていないこと」。 | 「検索のスコープは？ タイトルのみ？ 本文も含む？」 |

Example Mappingの力は、この4要素を**順番に書いていく過程**にあります。Storyを書き、Ruleを洗い出し、各Ruleに具体的なExampleをつけていくと、「あれ、この場合はどうなるんだろう？」というQuestionが自然に湧いてきます。

Questionが出てくるということは、仕様に「穴」があるということです。開発前にその穴を見つけられることが、Example Mappingの最大の価値です。

### 普通の仕様書との違い

従来の仕様書は「要件を文章で書く」ことが中心です。

```
従来の仕様書
============
お問い合わせ一覧画面
- お問い合わせの一覧を表示する
- ページネーション機能を備える
- ステータスでフィルタリングできる
```

これは「何を作るか」のリストです。各項目がどういう条件でどう動くかは読み手の解釈に委ねられています。

Example Mappingは「どう動くか」を具体例で示します。

```
Example Mapping
===============
Story: 管理者として、お問い合わせ一覧を見たい

Rule: 一覧はページネーションされる
  Example: 0件 → 「お問い合わせはありません」を表示
  Example: 1〜20件 → ページネーションなし、全件表示
  Example: 21件以上 → ページネーション表示、1ページ20件

Rule: ステータスでフィルタリングできる
  Example: 「未対応」選択 → 未対応のみ表示
  Example: 「対応済」選択 → 対応済のみ表示
  Example: 「全て」選択 → 全件表示
  Example: フィルタ適用後に0件 → 「条件に一致するお問い合わせはありません」を表示
```

具体例を書くことで、「フィルタ適用後に0件の場合はどうする？」という問いが自然に生まれます。従来の仕様書では見落としがちなエッジケースです。

## 実演1: お問い合わせ一覧（Webアプリ）

ここからは、実際にExample Mapを一歩ずつ構築していきます。最初の例は、冒頭で紹介した「お問い合わせ一覧」です。この実演を通じて、Example Mappingのプロセスそのものを体験してください。

### Step 1: Storyを定める

まず、何を作るかを1文で表現します。

```
Story: 管理者として、お問い合わせ一覧を見たい
  目的: 受信したお問い合わせの状況を把握し、対応の優先順位を判断する
```

ここで大事なのは「誰が」と「なぜ」です。「管理者として」と書くことで、権限の話が自然に出てきます。「状況を把握し、対応の優先順位を判断する」と書くことで、ソート順やステータス表示の重要性が見えてきます。

### Step 2: Ruleを洗い出す

Storyを成立させるために必要なビジネスルールを列挙します。

最初に思いつくものから書き始めます。完璧でなくて構いません。Exampleを書いていく過程で新しいRuleが見つかることがよくあります。

```
Rule 1: 一覧はページネーションされる
Rule 2: 一覧はデフォルトで新しい順に表示される
Rule 3: ステータスでフィルタリングできる
Rule 4: 各行をクリックすると詳細画面に遷移する
Rule 5: 管理者のみアクセスできる
```

### Step 3: 各RuleにExampleをつける

ここがExample Mappingの核心です。各Ruleに対して「こういう入力のとき、こう動く」という具体例を書きます。

```
Rule 1: 一覧はページネーションされる
─────────────────────────────────
  Example 1-A: お問い合わせが0件のとき
    → テーブルは表示されない
    → 「お問い合わせはまだありません」というメッセージが表示される

  Example 1-B: お問い合わせが1〜20件のとき
    → 全件がテーブルに表示される
    → ページネーションUIは表示されない

  Example 1-C: お問い合わせが21件以上あるとき
    → 最初の20件がテーブルに表示される
    → ページネーションUIが表示される
    → 「2ページ目」をクリックすると21〜40件目が表示される

  Example 1-D: お問い合わせが合計25件あるとき
    → 2ページ目に5件が表示される
    → 「次のページ」ボタンは表示されない（最終ページのため）
```

```
Rule 2: 一覧はデフォルトで新しい順に表示される
────────────────────────────────────────────
  Example 2-A: 3件のお問い合わせがある（作成日: 1/1, 1/5, 1/3）
    → 表示順は 1/5, 1/3, 1/1（降順）

  Example 2-B: 同じ日に複数のお問い合わせがある
    → 作成時刻の降順で表示される
```

```
Rule 3: ステータスでフィルタリングできる
──────────────────────────────────────
  Example 3-A: 「未対応」を選択
    → ステータスが「未対応」のお問い合わせのみ表示される

  Example 3-B: 「対応中」を選択
    → ステータスが「対応中」のお問い合わせのみ表示される

  Example 3-C: 「対応済」を選択
    → ステータスが「対応済」のお問い合わせのみ表示される

  Example 3-D: 「全て」を選択（デフォルト）
    → 全てのステータスのお問い合わせが表示される

  Example 3-E: フィルタ適用後に該当件数が0件
    → 「条件に一致するお問い合わせはありません」が表示される
    → フィルタの選択状態はそのまま維持される
```

```
Rule 4: 各行をクリックすると詳細画面に遷移する
────────────────────────────────────────────
  Example 4-A: ID "contact-123" の行をクリック
    → /contacts/contact-123 に遷移する

  Example 4-B: 行のクリック可能な範囲
    → 行全体がクリック可能（ステータスバッジ部分も含む）
```

```
Rule 5: 管理者のみアクセスできる
─────────────────────────────
  Example 5-A: 管理者がアクセス
    → 一覧が表示される

  Example 5-B: 一般ユーザーがアクセス
    → 403エラーが表示される
    → 「アクセス権限がありません」というメッセージ
```

### Step 4: Questionを洗い出す

Exampleを書いていく過程で、決まっていないことや確認が必要なことが浮かび上がってきます。これがQuestionです。

```
Question（未解決の疑問）
======================
  Q1: 検索機能は必要か？
    → 必要な場合、検索のスコープは？（タイトルのみ？ 本文も？ 送信者名も？）

  Q2: カラムのソートは変更可能にするか？
    → 例: ステータスカラムのヘッダーをクリックしてソート切替

  Q3: 一覧に表示するカラムは？
    → ID、タイトル、ステータス、作成日以外に何を表示するか

  Q4: 一括操作は必要か？
    → 例: チェックボックスで選択して一括ステータス変更

  Q5: ステータスの種類は3つ（未対応・対応中・対応済）で確定か？
    → 他にステータスを追加する可能性があるか
```

この5つのQuestionは、「お問い合わせ一覧を作って」という一行の指示からは絶対に出てきません。Example Mapを書く過程で初めて「あ、これ決まってないな」と気づけるのです。

### Step 5: Questionに回答し、新しいRuleを発見する

Questionに答えることで、新しいRuleが生まれることがあります。

| Question | 回答 | 結果 |
|---|---|---|
| Q1: 検索機能は必要か？ | 今回のスコープに含める。タイトルのみを検索対象とする。 | 新しいRule 6を追加 |
| Q2: カラムのソートは変更可能にするか？ | 今回は不要。将来的に検討。 | Ruleの追加なし |
| Q3: 一覧に表示するカラムは？ | ID、タイトル、送信者名、ステータス、作成日の5カラム。 | Rule 2の補足として記録 |
| Q4: 一括操作は必要か？ | 今回は不要。 | Ruleの追加なし |
| Q5: ステータスの種類は確定か？ | 確定。ただし将来の拡張を考慮して、ステータスはマスタデータとして管理する。 | 実装上の注意事項として記録 |

新しく見つかったRule 6にもExampleをつけます。

```
Rule 6: タイトルで検索できる
──────────────────────────
  Example 6-A: 「返金」で検索
    → タイトルに「返金」を含むお問い合わせのみ表示される

  Example 6-B: 「存在しないキーワード」で検索
    → 0件表示（「条件に一致するお問い合わせはありません」）

  Example 6-C: 検索とフィルタを同時に使用
    → 両方の条件を満たすお問い合わせのみ表示される
    → 例: ステータス「未対応」かつタイトルに「返金」を含む

  Example 6-D: 検索キーワードをクリア
    → フィルタの状態は維持したまま、全件（またはフィルタ適用結果）に戻る
```

### Before/After: 一行指示 vs Example Map付き指示

ここまでの作業の結果を、AIへの指示として比較してみます。

**Before（一行指示）**

```
お問い合わせ一覧画面を作ってください。
ステータスでフィルタリングできるようにしてください。
```

**After（Example Map付き指示）**

```
お問い合わせ一覧画面を作ってください。
以下のルールと具体的な動作を参考にしてください。

【ページネーション】
- 0件のとき: テーブル非表示、「お問い合わせはまだありません」を表示
- 1〜20件のとき: 全件表示、ページネーションUIなし
- 21件以上のとき: 1ページ20件でページネーション表示

【ソート】
- デフォルトは作成日時の降順（新しい順）

【フィルタリング】
- ステータス: 「全て」（デフォルト）/ 「未対応」/ 「対応中」/ 「対応済」
- フィルタ適用後に0件の場合: 「条件に一致するお問い合わせはありません」を表示

【検索】
- タイトルのみ対象
- フィルタとの併用可能
- 検索クリア時はフィルタ状態を維持

【表示カラム】
- ID、タイトル、送信者名、ステータス、作成日

【遷移】
- 行クリックで /contacts/{id} に遷移

【権限】
- 管理者のみアクセス可能
- 一般ユーザーがアクセスした場合は403エラー

【対象外（今回は実装しない）】
- カラムソートの切替
- 一括操作
```

同じ「お問い合わせ一覧」を作るための指示ですが、情報量がまるで違います。Afterの指示であれば、AIが「一般的にこうだろう」と補完する余地がほとんどありません。これが15件の仕様齟齬を防ぐ構造です。

言語を超えた教訓: 仕様の抜け漏れは「書いていないこと」から生まれる。Example Mappingは「書くべきこと」を構造的に洗い出す道具である。

## 実演2: REST API エンドポイント（バックエンド）

Example Mappingは画面系の機能だけでなく、APIの設計にも使えます。ここでは「ユーザー登録API」を例に実演します。

### Story

```
Story: 新規ユーザーとして、アカウントを作成したい
  目的: サービスの機能を利用するためにアカウントを作成する
  エンドポイント: POST /api/users
```

### Ruleの洗い出し

```
Rule 1: リクエストボディのバリデーション
Rule 2: メールアドレスの重複チェック
Rule 3: パスワードのハッシュ化
Rule 4: 確認メールの送信
Rule 5: レスポンスの形式
```

### 各RuleのExample

```
Rule 1: リクエストボディのバリデーション
──────────────────────────────────────
  Example 1-A: 正常なリクエスト
    Input:  { "email": "user@example.com", "password": "SecureP@ss1", "name": "田中太郎" }

  Example 1-F: nameが空
    Input:  { "email": "user@example.com", "password": "SecureP@ss1", "name": "" }
    Output: 400 Bad Request
    Body:   { "error": "validation_error", "details": [{"field": "name", "message": "名前は必須です"}] }

  Example 1-G: 複数のバリデーションエラーが同時に発生
    Input:  { "email": "", "password": "short", "name": "" }
    Output: 400 Bad Request
    Body:   { "error": "validation_error", "details": [
                {"field": "email", "message": "メールアドレスは必須です"},

```

    → レスポンスにパスワードフィールドは含まれない

  Example 3-B: 同じパスワードでもハッシュ値は毎回異なる（salt付き）
```

  Example 4-A: 正常に登録完了した場合
    → 登録メールアドレス宛に確認メールが送信される
    → メールには確認用URLが含まれる
    → 確認用URLの有効期限は24時間

    → ユーザー登録自体は成功する（ロールバックしない）

```

======================

  Q2: ユーザー名の文字数制限は？（最小・最大）
  Q3: レートリミットは必要か？（連続登録攻撃の防止）

このExample Mapを見ると、「ユーザー登録APIを作って」という一言では伝えきれない仕様が大量にあることがわかります。特にExample 2-C（メールアドレスの大文字小文字）やExample 4-B（メール送信失敗時の挙動）は、Example Mapを書かなければ見落としていた可能性が高いケースです。

言語を超えた教訓: APIの仕様は「正常系」だけでなく「異常系の振る舞い」を具体例で定義することが、齟齬を防ぐ鍵である。

## 実演3: CLIツール（言語非依存）

3つ目の実演として、CLIツールの仕様をExample Mapで整理します。CLIツールはWebアプリやAPIと比べて画面がない分、「入力と出力」が明確です。それでもExample Mappingを使うと、意外な抜け漏れが見つかります。

### Story

Story: 開発者として、CSVファイルをJSON形式に変換したい
  目的: 既存のCSVデータをアプリケーションで扱いやすいJSON形式に変換する
  コマンド: csv2json <input-file> [options]

## 抽象原則: Example Mappingは言語に依存しない

3つの実演（Webアプリ、REST API、CLIツール）を通じて、Example Mappingの適用範囲の広さを示しました。ここで、その抽象的な原則を整理します。

### 「何を作るか」と「どう作るか」を分離する技術

Example Mappingで定義するのは「何を作るか（What）」です。「どう作るか（How）」は含みません。

お問い合わせ一覧のExample Mapは、実装言語がPythonでもTypeScriptでもGoでもRustでも同じです。ページネーションのルール、0件時の表示、権限制御の仕様は、使うフレームワークやデータベースに関係なく成り立ちます。

```
同じExample Mapから生まれる実装の違い
====================================

Example Map（言語非依存）:
  Rule: 0件のとき「お問い合わせはまだありません」を表示

Python (FastAPI):
  @app.get("/contacts")
  async def list_contacts():
      contacts = await contact_repo.find_all()
      if not contacts:
          return {"message": "お問い合わせはまだありません", "data": []}
      ...

TypeScript (Next.js):
  export default function ContactList({ contacts }: Props) {
    if (contacts.length === 0) {
      return <EmptyState message="お問い合わせはまだありません" />;
    }
    ...
  }

Go (net/http):
  func listContacts(w http.ResponseWriter, r *http.Request) {
      contacts, _ := repo.FindAll(r.Context())
      if len(contacts) == 0 {
          json.NewEncoder(w).Encode(map[string]any{
              "message": "お問い合わせはまだありません",
              "data":    []any{},
          })
          return
      }
      ...
  }
```

実装は言語ごとに異なりますが、仕様は同じです。AIに渡すExample Mapが同じであれば、どの言語で実装しても同じ振る舞いになります。

### AIとの協業で最も重要なこと

AIとの協業では、「コードを書くこと」はAIが得意です。「何を作るか」を決めることは、人間にしかできません。

| 責任領域 | 内容 |
|---|---|
| **人間が決めること** | ユーザーストーリー（誰が、何のために使うのか） / ビジネスルール（どういう条件で、どう動くか） / 具体的な動作例（こういう入力のとき、こう動く） / 未解決の疑問の特定と回答 |
| **AIが担うこと** | Example Mapに基づく実装 / テストコードの生成 / エッジケースの検出（人間が見落とした場合の補完） / リファクタリングと最適化 |

Example Mappingは、この役割分担を構造化する道具です。人間が「何を作るか」を明確にすればするほど、AIは「どう作るか」に集中できます。

### Example Mapの品質チェックリスト

Example Mapを書いたら、AIに渡す前に以下をチェックしてみてください。

- 各Ruleに最低2つのExampleがあるか？（正常系 + 異常系/エッジケース）
- 0件/空/null のケースを考慮しているか？
- 権限やアクセス制御に関するRuleがあるか？
- エラー時のユーザーへの表示を定義しているか？
- Questionが3つ未満なら、本当に全て決まっているか再考する
- 「対象外」（今回実装しないこと）を明記しているか？

Questionが0個のExample Mapは、むしろ疑ったほうがいいです。仕様に完全に穴がないことは稀で、多くの場合は「考えが足りていない」サインです。

## SFADでの実装

少しだけ触れますが、私はこのExample MappingのプロセスをClaude Code Skill（`/sfad cycle`）として自動化しました。

コマンドを実行すると、対話形式でStory → Rule → Example → Questionを順番に引き出し、構造化された仕様書を生成します。手動でExample Mapを書くと30分以上かかる作業が、AIとの対話で10〜15分に短縮されました。

重要なのは、Example Mappingの「思考プロセス」は人間が行い、「構造化と整形」をAIが支援するという役割分担です。「何を作るか」を考えるのは人間。その考えを漏れなく整理するのをAIが手伝う。このバランスが、仕様齟齬15件を構造的に潰す鍵でした。

詳細はSFADシリーズの別記事で紹介しています。

## まとめ

15件の仕様齟齬を振り返って、見えてきたことを整理します。

「お問い合わせ一覧を作って」という一行の指示は、私にとっては「あの画面」のイメージが鮮明に浮かんでいました。でもAIにとっては「お問い合わせ一覧という名前の何か」でしかありませんでした。頭の中にある仕様を構造化して伝えなければ、AIは推測で補完するしかありません。その推測が私の意図と一致する確率は、残念ながら高くありません。

Example Mappingは、この問題を「気合いで詳しく書く」のではなく、「構造に沿って網羅的に洗い出す」方法で解決します。Story → Rule → Example → Questionという4つの箱を順番に埋めていくだけで、自分が書き忘れていた仕様が次々と見つかります。

そしてこの手法は、Webアプリでも、REST APIでも、CLIツールでも、Python・TypeScript・Go・Rust、どの言語で実装する場合でも同じように使えます。仕様の構造は言語に依存しないからです。

仕様が決まったら、次に必要なのはテスト戦略です。Example Mapで「何を作るか」を定義したら、次は「作ったものが壊れないこと」を保証する仕組みが必要です。
