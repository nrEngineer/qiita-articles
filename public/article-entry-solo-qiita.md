Claude CodeでTDDを回し始めてから、デグレが0になりました。

以前は深夜2時に「動いたからマージしよう」とプッシュして、3日後にクライアントから「XXが動かないんですが」と連絡が来る、という生活でした。テストは書いていない。レビューしてくれる人もいない。謝りながら修正して、夜中に再デプロイする。

これ、あなたの話じゃないですか？

私の話です。

---

## なぜ1人開発で品質が崩壊するのか

1人開発の品質問題は、「サボっているから」ではありません。**フィードバックループがないから**です。

チーム開発には、品質を保つ仕組みが自然に備わっています。コードレビューがある。CIが走る。誰かが「これ大丈夫?」と聞いてくれる。問題が外から見えるから、直す機会がある。

1人開発にはこれがない。

| チーム開発 | 1人開発 |
|-----------|---------|
| レビュアーがいる | 自分だけ |
| CIが走る | 手動 or なし |
| 誰かが指摘してくれる | 誰も言わない |
| マージ前にチェックが入る | 「動いたらOK」 |

誰も怒らないから、テストを書かない。誰も見ないから、lint設定が甘いまま。誰も気づかないから、AIが書いたコードをそのままマージする。これは個人の問題ではなく、**仕組みの不在**という問題です。

### 「動く」と「正しい」の区別がつかない

AIと開発していると、もう一つ厄介な問題が出てきます。AIが書くコードは、見た目がきれいです。インデントが揃っている。命名が一貫している。構文エラーがない。フォーマッタを通したかのように整っている。

でも「見た目がきれい」は「正しい」を意味しません。

AIは「仕様を知らない」からです。「ユーザー一覧を返して」と言えば、全件返すコードを書く。ページネーションが必要だということは、言わなければわからない。ページ20件で返すのか50件なのか、ソートはどの順序なのか、削除済みユーザーは含めるのか——AIはこれらを推測するしかない。

推測で書かれたコードは、「動く」かもしれないが「正しい」とは限らない。レビュアーがいれば「これ、削除済みユーザーも入ってない?」と気づいてくれる。いなければ、本番で初めてわかる。

## 解決策1: 仕組みをレビュアーにする

「レビュアーがいない」問題の答えは、「レビュアーを雇う」ではありません。**仕組みをレビュアーにする**ことです。

lint strictとCIは、疲れません。休みません。全コミットに対して同じ基準でチェックします。しかも一度設定すれば、半永久的に動き続けます。

SFADのバグ分析で見えた数字があります。78件のバグのうち、**lint strictだけで35件（45%）が自動検出可能**でした。最多カテゴリのprint残留（23件）とbare except（12件）は、ほぼ全件がlintで機械的にブロックできる問題でした。

設定に30分かければ、翌日から「無人レビュー」が動き始めます。

### 3言語の最小構成

| 言語 | lintツール | 型チェック | 主要ルール |
|------|-----------|------------|-----------|
| Python | ruff | mypy --strict | T201(print), E722(bare except), ANN401(Any) |
| TypeScript | ESLint | tsc --noEmit | no-console, no-empty, strict: true |
| Go | golangci-lint | go vet | forbidigo(fmt.Print禁止), errcheck |

各言語の具体的な設定ファイルは、バグ78件の分析をもとに組み立てた完全版をArt.1に公開しています。既存コードを除外しながら段階的に有効化する方法も含めてコピペ可能な状態にまとめているので、設定ファイルが手元にない方はそちらを参照してください。

→ **Art.1「78バグの全分類」**（Qiita、無料）: 完全な設定ファイルあり

### pre-commit hookで「マージ前に最低限」を強制する

lintを手動で実行していると、急いでいるときに飛ばします。人間はそういうものです。pre-commit hookに入れると、コミット時に自動で走ります。

**Python**: `pip install pre-commit && pre-commit install` の後、`.pre-commit-config.yaml` に `ruff-pre-commit` を追加するだけです。

**TypeScript**: `husky` + `lint-staged` の組み合わせ。`npx husky init` の後、`package.json` の `lint-staged` に `["eslint --fix", "tsc --noEmit"]` を追加します。

**Go**: `golangci-lint` を `.git/hooks/pre-commit` に書くか、`pre-commit` フレームワークの公式 hook を使います。

## 解決策2: AIを「ペア」ではなく「プロセス」にする

AIとの開発でよくある使い方は「ペアプログラミング」的なものです。「これ実装して」「直して」「次はこれ」と、会話しながら進める。

この使い方には根本的な問題があります。**AIは仕様を知らないのに、仕様を推測して実装する**という問題です。

解決策は、AIに「何を作るか」を先に渡すことです。仕様が決まってから実装する。これがSFADの核心である「Spec-First」の考え方です。

### Example Mappingで仕様を先に決める

Example Mappingは、3つの要素で仕様を構造化する手法です。

| 要素 | 内容 | 例 |
|------|------|-----|
| ルール | その機能のビジネスルール | 「1ページ20件で表示する」 |
| 具体例 | ルールが適用されるケース | 「100件中1ページ目: 1〜20件が表示される」 |
| 疑問点 | まだ決まっていないこと | 「削除済みユーザーは含めるか?」 |

実際に書いてみると、「まだ決まっていないこと」が必ず出てきます。この疑問点を解消してから実装に入る。それだけで仕様の認識齟齬は大幅に減ります。

私の分析では、78件中15件が「仕様の認識齟齬」でした。そのほぼ全件が、Example Mappingで事前に解消できた問題でした。

```markdown
## ユーザー登録 - Example Mapping

### ルール
- メール必須、パスワード8文字以上、重複メールはエラー

### 具体例
- 正常系: 有効なメール + 8文字以上 → 登録成功、確認メール送信
- エラー: 既存メール → 「このメールアドレスは使用済みです」
- エラー: パスワード7文字 → 「パスワードは8文字以上必要です」

### 疑問点（→ 解決）
- ソーシャルログインは? → v1では不要
- メールの大文字小文字は区別するか? → 区別しない（小文字正規化）
```

この仕様をAIに渡す。そうすると、推測なしに実装できます。

### AIがPO/Dev/QA 3役を担う: `/sfad cycle`

SFADには`/sfad cycle`というコマンドがあります。機能を実装するとき、AIが3つの視点で仕様を整理してから実装に進みます。

- **PO（プロダクトオーナー）視点**: ビジネス要件の確認、受け入れ条件の定義
- **Dev（開発者）視点**: 実装の設計、テストケースの洗い出し
- **QA（品質保証）視点**: エッジケースの検討、潜在的な問題の指摘

1人開発では、この3役を自分でやらなければなりません。でも「急いでいる」「疲れている」「プレッシャーがある」という状況では、QA視点が抜けがちです。AIにこの3役をやらせると、人間が見落とした視点を補完してくれます。

コマンドを呼ぶと、AIはまずDiscovery（発見）フェーズで3視点から仕様を整理します。人間がそれを確認・承認してから実装に入る。このゲートがあるだけで、「動いたからマージ」の衝動を抑制できます。

### 自分専用レビュアーSkillを作る

Claude Code Skillとは、AIに「特定のタスクの実行方法」を教えるMarkdownファイルです。コードを書く必要はありません。チェックリストと手順を書くだけです。

1人開発者にとって最も効果的なSkillの1つが「コードレビューSkill」です。`~/.claude/skills/review/review.md` に以下のようなファイルを作ります。

```markdown
# コードレビューの手順

## Phase 1: 品質チェック
- [ ] print文/console.log/fmt.Printlnが残っていないか
- [ ] bare except/空のcatchブロックがないか
- [ ] Any/any/interface{}が使われていないか
- [ ] ユーザー入力がSQL文に直接結合されていないか

## Phase 2: 仕様整合性チェック
- [ ] Example Mappingで定義したルールと実装が一致しているか
- [ ] 境界値のケース（0件、1件、上限件数）が処理されているか

## Phase 3: テストチェック
- [ ] 正常系・エラー系・境界値のテストがあるか

## Gate
全チェック通過時のみ「実装完了」と報告する。
問題があれば、何がどう問題かを具体的に指摘する。
```

`/review` コマンドで呼び出す。毎回同じチェックリストで、同じ基準でレビューが走ります。疲れても、急いでいても、AIはチェックリストを飛ばしません。これが「レビュアーを仕組みにする」という意味です。

このSkillのポイントは「チェックリストの粒度」です。「バグがないか確認して」という曖昧な指示では、AIも曖昧なチェックしかできません。`bare except がないか` `SQL文に変数を直接結合していないか` という具体的な指示があってはじめて、機械的なチェックが機能します。

自分のプロジェクトで実際に踏んだバグをチェックリストに追加していくのが、このSkillを育てる一番の方法です。SFADではバグが出るたびに `review.md` に一行追加します。同じミスを2度繰り返さないために。

## 1人開発者の週間ワークフロー

「理論はわかったけど、実際の1週間はどう動けばいい?」という疑問に答えます。

### 月曜: 今週の機能をExample Mappingで定義する（30〜60分）

週の最初に「今週何を作るか」を定義します。Example Mappingのテンプレートを使って、ルール・具体例・疑問点を書き出す。疑問点はクライアントに確認するか、自分で仕様を決める。決まった仕様を `docs/spec/[機能名].md` に保存する。これがAIへの入力になります。

### 火〜木: cycleとimplでテスト駆動実装（毎日2〜4時間）

仕様が決まったら実装に入ります。

1. `/sfad cycle user-registration` でDiscoveryを実行（AIが3視点で仕様を整理）
2. AIの提案を確認・承認する（Gate）
3. `/sfad impl user-registration` で実装
4. `pytest tests/test_user_registration.py -v` でテスト確認

AIが「これでよいですか?」と聞いてくるゲートが数回あります。この確認を飛ばさないことが重要です。急いでいるときほど飛ばしたくなります。でも飛ばすと「動いたからマージ」の地獄に戻ります。

### 金曜: lint + テスト全通し + デプロイ（1〜2時間）

```bash
# lint全チェック（言語に合わせて選択）
ruff check . && mypy . --strict
npx eslint . && npx tsc --noEmit
golangci-lint run

# テスト全通し
pytest        # Python
npm test      # TypeScript
go test ./... # Go
```

lint警告が0件、テストが全通しの状態でのみデプロイする。これをルールにする。例外を作らない。「今週は忙しかったから」は例外にしない。

## 「仕組み」は裏切らない

1人開発の孤独は本物です。深夜、誰も見ていない中でコードを書く。「これで正しいのか」という不安は消えない。レビューしてくれる人がいたら、という気持ちはあります。

でも仕組みは裏切りません。

pre-commit hookは、あなたが疲れていても動きます。CIは、あなたが急いでいても走ります。Example Mappingのテンプレートは、あなたが考えるべき疑問点を思い出させてくれます。Skillは、あなたが見落としたチェック項目を指摘してくれます。

**1人だからこそ、仕組みが必要です。** チームには人が補ってくれる。1人には仕組みしかない。

Day 0に30分かけてlintを入れる。仕様を書いてからAIに渡す。自分専用のレビューSkillを作る。この3つだけで、「動いたからマージ」から「確認してからデプロイ」に変わります。
