# Claude Code Skillの作り方 完全ガイド ― 開発手法をAIに覚えさせて、コマンド1つで再現する

最初に書いたSkillファイルは、完全に無視されました。

Claude Codeはファイルを読み、「了解しました」と答え、そして指示と全く関係ない実装を始めました。まるで何も読んでいないかのように。

何が間違っていたのか。3日間、20回以上書き直して、ようやくわかりました。私は「手順書」を書いていたのです。AIに必要だったのは「判断基準」でした。

「Step 1: テストを書く。Step 2: 実装する」── こんな指示は、AIにとって情報量ゼロです。AIはすでにテストの書き方も実装の仕方も知っています。必要なのは「どんなテストを、どの粒度で、何を基準に書くか」という判断の基準でした。

この発見から、私のSkill設計は根本的に変わりました。

---

## この記事でしか読めないこと

- **Skill設計の7ステップ方法論**: 「手法の言語化→構造化→検証→改善」の具体的プロセス
- **3つの実践例**: SFAD以外にも応用できる、コードレビュー・デプロイ・ドキュメント生成のSkill設計
- **よくある失敗パターンと回避策**: 20回以上の書き直しから学んだ「AIが無視するSkill」の特徴
- **完全なSkillファイルのテンプレート**: コピペして自分の手法を載せるだけで動く骨格
- **ゲート設計の実装例**: 「手順を飛ばさせない」仕組みの具体的な書き方

## 「手順を知っている」と「手順が動く」の違い

ある業務システムの開発で78件のバグを踏んだ後、私は7つの設計原則を導きました。原則をドキュメントに書きました。チームに共有しました。

1ヶ月後、同じパターンのバグが出ました。

ドキュメントは読まれていませんでした。正確に言うと、読んだ人はいました。でも急いでいるときに参照する人はいませんでした。プレッシャーがかかると、「今回だけは手順を飛ばしても大丈夫だろう」という判断をしてしまうのです。これは人間の性質であって、個人の怠慢ではありません。

ここで私は気づきました。「手順を知っている」と「手順が動く」は全く別のことだと。

**ドキュメントに書いた手法**:

- 読まれない（忙しいから）
- 読まれても守られない（判断を人に委ねているから）
- 守られたとしても人によって解釈が違う（曖昧さが残るから）

**Skillにした手法**:

- コマンド1つで手法が起動する（読む必要がない）
- AIがゲートを守る（手順を飛ばせない）
- 手順が属人化しない（同じコマンドで同じ動きをする）

「開発手法をコードにする」という発想。これがClaude Code Skillの本質です。

ドキュメントは「知識を伝える」ためのもの。Skillは「手順を実行させる」ためのもの。この違いを理解すると、Skillの設計方針が見えてきます。

## Claude Code Skillの基礎知識

### Skillとは何か

Claude Code Skillとは、AIに「特定のタスクの実行方法」を教えるMarkdownファイルです。

Claude CodeはAnthropicが公式に提供するCLIツールです。このCLIには、ユーザーが独自のスキルとコマンドを定義して拡張できる仕組みがあります。コードを書く必要はありません。手順をMarkdownで言語化するだけです。

### 2層構造

Skillは2つのディレクトリで構成されます。

```
~/.claude/skills/{name}/   ← スキルロジック（AIが参照する詳細手順）
~/.claude/commands/{name}/ ← コマンドインターフェース（ユーザーが呼び出す）
```

この分離が重要です。

**commands/ は「窓口」です**。ユーザーが `/コマンド名` と入力したときの入口になります。短くていいです。「何をするか」「どのスキルに従って動くか」だけ書きます。

**skills/ は「脳みそ」です**。AIが実際に従う手順、判断基準、チェックリスト、エラー処理をすべて書きます。長くなっても構いません。AIはこのファイルを読んで、書かれた通りに動きます。

### ファイルの役割

一つのSkillは、通常以下のファイルで構成されます。

- **SKILL.md**: スキル全体のメタ情報、サブスキル一覧、理論的基盤。スキルの「設計書」に相当します
- **サブスキルファイル（例: cycle.md, spec.md）**: 各コマンドの詳細な実行手順。スキルの「実行マニュアル」に相当します
- **コマンドファイル（commands/内）**: ユーザーが呼び出すエントリポイント。スキルの「窓口」に相当します

## SFADの実際のファイルから学ぶ

理論だけでは伝わらないので、私が実際に作ったSFAD（Spec-First AI Development）のファイルを使って説明します。

### SKILL.md の構造

SFADのSKILL.mdを簡略化して示します。

```markdown
---
name: sfad
description: "Spec-First AI Development (SFAD) - BDD Discovery + Double-Loop TDD を統合した、AI時代の仕様駆動開発フレームワーク"
---

# SFAD: Spec-First AI Development

## Theoretical Foundation

- Canon TDD（Kent Beck, 2023）: Test List → Pick → Red → Green → Refactor
- BDD 3 Practices（Seb Rose & Gaspar Nagy）: Discovery → Formulation → Automation
- Double-Loop TDD（Freeman & Pryce, GOOS）: 外側ループ（受け入れ）+ 内側ループ（ユニット）
- Example Mapping（Matt Wynne）: Story → Rules → Examples → Questions

## Sub-Skills

- /sfad cycle — BDD Discovery + Double-Loop TDD 全自動サイクル（メイン推奨）
- /sfad spec — 仕様定義のみ手動実行
- /sfad test — テスト設計のみ手動実行
- /sfad impl — 実装のみ手動実行
- /sfad reverse — 既存コードから仕様を逆算
- /sfad init — プロジェクト初期化

## Usage

/sfad cycle "お問い合わせ管理"
/sfad cycle "ユーザー認証" --be
/sfad reverse @src/services/order_service.py

## Auto-Detection

Stack Detection Matrix:
- Python: pytest, ruff, mypy
- TypeScript: vitest, eslint, tsc --strict
- Go: go test, golangci-lint, go vet
（言語を自動検出し、対応するツールチェーンを選択）

## Principles

1. 仕様が先、コードは後
2. テストは仕様の実行可能な表現
3. Day 0に品質基盤を構築
4. Discovery → Formulation → Automation の順序を守る
5. ゲートで人間の判断を介在させる
6. 仕組みで防ぐ、規律に頼らない
7. 複利を意識する（早く入れるほど効果が大きい）
```

### なぜこの構造にしたか

SKILL.mdの各セクションには、それぞれ明確な理由があります。

**`name` + `description`（フロントマター）**

AIがスキルを認識するためのメタデータです。`description`には、ユーザーが入力しそうなキーワードを含めます。「BDD」「TDD」「仕様駆動」など、このスキルが呼ばれるべきシチュエーションで使われる言葉を入れておくと、AIが適切なスキルを選択できます。

最初は`description`を短くしていましたが、AIがスキルの適用場面を判断しにくいことがわかり、キーワードを増やしました。

**`Theoretical Foundation`（理論的基盤）**

AIが「なぜこの手順なのか」を理解するためのセクションです。手順だけ書くと、AIは手順を実行しますが、ユーザーに「なぜこうするのか」を聞かれたときに答えられません。

理論的基盤を書いておくと、AIは「このPhaseはBDDのDiscoveryに対応しています」と説明できるようになります。ユーザーの理解を助ける効果があります。

**`Sub-Skills`（サブスキル一覧）**

親子関係を定義するセクションです。AIが「ユーザーはどのサブコマンドを実行したいのか」を判断するために必要です。

一覧がないと、AIは「cycle.mdを使うべきか、spec.mdを使うべきか」を迷います。迷ったAIは、独自の判断で動きます。それはたいてい期待と違う動きになります。

**`Auto-Detection`（自動検出マトリクス）**

言語やフレームワーク固有の実装を自動選択するためのセクションです。SFADは10スタックに対応しているので、「Pythonならpytest、TypeScriptならvitest」という対応を定義しています。

これがないと、ユーザーが毎回「Pythonを使っています」と明示する必要があります。プロジェクトのファイルを見て自動判断できる方が便利です。

**`Principles`（設計原則）**

AIの判断基準です。Phaseの実行中にAIが迷ったとき、原則に立ち返って判断します。

例えば「テストを書く前に実装を始めてもいいか？」という判断が生じたとき、原則2「テストは仕様の実行可能な表現」と原則4「Discovery → Formulation → Automation の順序を守る」に照らして「いいえ」と判断できます。

原則がないと、AIは「効率を考えると実装を先にした方が早い」という判断をすることがあります。それは正しい判断かもしれませんが、SFADの設計意図とは異なります。

### cycle.md の構造

SFADの中核であるcycle.mdを簡略化して示します。

```markdown
# SFAD Cycle — BDD Discovery + Double-Loop TDD 全自動サイクル

## Theoretical Flow

BDD 3 Practices → SFAD Cycle Phase → TDD Loop の対応:

- Discovery → Phase 1-2 → Example Mapping + Given-When-Then
- Formulation → Phase 3 → 受け入れテスト生成（外側ループ）
- Automation → Phase 4-8 → Test List → Unit Test → Impl → Refactor → Accept

## Execution Flow

### Phase 1: Discovery — Example Mapping
1. AI が3つの視点（ハッピーパス/異常系/境界値）で Example Map を作成する
2. ドラフトを以下の形式でユーザーに提示する:
   - Story: 1行で機能の目的を書く
   - Rules: ビジネスルールを箇条書きで列挙する
   - Examples: 各ルールの具体例を書く
   - Questions: 未確定事項をリストアップする
3. ゲート: ユーザーの回答を待つ。勝手に先に進まない
4. Questions が 0 になるまで繰り返す

### Phase 2: Formulation — Given-When-Then 変換
1. 確定した Example を Given-When-Then 形式に変換する
2. ハッピーパスと異常系を明示的に分類する
3. ゲート: ユーザーにGiven-When-Thenの内容を確認してもらう

### Phase 3: Automation — 受け入れテスト生成
1. Given-When-Then から受け入れテストコードを生成する
2. テストを実行して、全て Red（失敗）であることを確認する
3. ゲート: 全テストが Red であることをユーザーに報告する

### Phase 4: Test List 作成
1. 受け入れテストを通すために必要なユニットテストをリストアップする
2. 優先順位をつける（ハッピーパスが先）

### Phase 5: ユニットテスト生成
1. Test List の先頭から1件ずつテストコードを生成する
2. 実行して Red であることを確認する

### Phase 6: 実装（Red → Green）
1. テストを1件ずつ通す
2. 最小限のコードで実装する（仮実装 → 三角測量）
3. 全ユニットテストが Green になるまで繰り返す

### Phase 7: リファクタリング
1. テストが全て Green の状態を維持しながらコードを整理する
2. 重複の除去、命名の改善、構造の整理

### Phase 8: 受け入れテスト確認
1. Phase 3 で生成した受け入れテストを再実行する
2. 全て Green であることを確認する
3. ゲート: 完了報告をユーザーに提示する

## ゲートとユーザー介入ポイント

- Phase 1 完了時: Example Map のレビュー（必須）
- Phase 2 完了時: Given-When-Then のレビュー（必須）
- Phase 3 完了時: 全テスト Red の確認（報告のみ）
- Phase 6 → Phase 7: 全テスト Green の確認（自動判定）
- Phase 8 完了時: 完了報告（必須）

## エラー時の振る舞い

- テスト実行が失敗（環境エラー）: エラーメッセージを表示して停止。ユーザーに環境確認を依頼する
- Phase 6 で3回連続 Green にならない: 現在のアプローチを中断し、ユーザーに方針を確認する
- ファイルが見つからない: 対象ファイルのパスをユーザーに確認する
- 想定外のプロジェクト構造: Auto-Detection の結果を表示し、ユーザーに確認する
```

### なぜこの構造にしたか

cycle.mdの各セクションにも、それぞれ理由があります。

**`Theoretical Flow`（理論的な流れ）**

AIが全体の地図を持つためのセクションです。8つのPhaseを個別に見ると「なぜこの順番なのか」がわかりにくくなります。BDDの3つのプラクティスとの対応を示すことで、AIは「今Phase 3にいるのは、BDDのFormulationが終わってAutomationに入ったところだ」と理解できます。

この理解があると、ユーザーに進捗を説明するときも「BDDのDiscoveryが完了しました。次はFormulationに入ります」と文脈のある説明ができます。

**各Phase（実行手順）**

「AIが何をするか」を具体的に定義しています。曖昧さを徹底的に排除しました。

最初のバージョンでは「Example Mappingを実施する」とだけ書いていました。すると、AIは独自の解釈でExample Mappingを行い、Questionsを出さずに次に進んでしまいました。「3つの視点で作成する」「Questionsが0になるまで繰り返す」と具体的に書いたら、期待通りに動くようになりました。

Skillの品質は「どれだけ具体的に書いたか」で決まります。

**ゲート（ユーザー介入ポイント）**

AIが暴走せず、人間の判断を仰ぐポイントを明示的に定義しています。

ゲートを入れなかったバージョンでは、AIはPhase 1からPhase 8まで一気に走りました。仕様の確認もなく、テストがRedかどうかの報告もなく、最後に「完了しました」と表示されました。結果は使い物にならないコードでした。

ゲートは「AIを止める」ためではなく「判断の質を担保する」ためにあります。仕様の確認を人間が行うことで、AIの解釈のズレを早期に修正できます。

**エラー処理（異常時の振る舞い）**

AIが行き詰まったときの行動を明確に定義しています。

エラー処理を書かなかったバージョンでは、テストが通らないときにAIが延々とリトライを続けました。10回目のリトライで「やはり通りません」と報告されても、最初のエラーメッセージはもう画面から消えています。

「3回連続Green にならない場合は停止してユーザーに確認」と定義したら、無駄なリトライがなくなりました。

ここまでがSkillの「読み方」です。2層構造、フェーズ設計、ゲート、エラー処理 ── 既存のSkillファイルを読んで理解するための知識は、ここまでで揃いました。

ここからは「作り方」です。あなた自身の開発手法 ── コードレビュー、デプロイ、テスト、ドキュメント生成、何でも ── をClaude Code Skillにする7ステップを、3つの実践例（コードレビュー・APIドキュメント生成・デプロイ自動化）とコピペ可能なテンプレートとともに解説します。

## 自分のSkillを0から作る7ステップ

最初から完璧なSkillを作る必要はありません。

### Step 1: ワークフローを言語化する

紙でもテキストファイルでも構いません。「私がこのタスクをやるとき、何をどの順番でやっているか？」を書き出します。

例として、コードレビューを考えてみます。

```
私がコードレビューをするとき:
1. PRの差分を開いて、変更の全体像を把握する
2. 変更されたファイルの一覧を確認する
3. セキュリティ的に問題がないかチェックする
4. パフォーマンスに影響しそうな箇所を探す
5. テストが書かれているか、カバレッジは十分か確認する
6. コード品質（命名、構造、重複）をチェックする
7. レビューコメントをまとめて投稿する
```

完璧でなくていいです。「普段やっていること」をそのまま書くだけです。書いてみると「あ、実はこの手順を飛ばしていることがあるな」と気づくことがあります。それも書いておきます。

### Step 2: Phaseに分割する

書き出した手順を、意味のあるまとまりに分けます。

分け方のポイントは3つあります。

- 独立して完結する工程で区切る
- 並列に実行できるものをグループ化する
- 依存関係を整理する（「AがないとBができない」を明確にする）

コードレビューの例で分けてみます。

```
Phase 1: スコープ特定（Step 1-2）
  → PR差分の読み込みと変更ファイルの一覧化
  → 他のPhaseの前提になるので最初に実行

Phase 2: セキュリティチェック（Step 3）
  → OWASPトップ10に照らしたチェック
  → Phase 3, 4と並列実行可能

Phase 3: パフォーマンス分析（Step 4）
  → N+1クエリ、メモリリーク、不要な再レンダリング
  → Phase 2, 4と並列実行可能

Phase 4: コード品質チェック（Step 6）
  → SOLID原則、命名規約、重複コード
  → Phase 2, 3と並列実行可能

Phase 5: テストカバレッジ確認（Step 5）
  → テストの存在確認と網羅性の評価
  → Phase 2-4の結果を踏まえて実行

Phase 6: レビューまとめ（Step 7）
  → 全Phaseの結果を統合してレビューコメントを生成
  → 全Phaseの完了が前提
```

「セキュリティ」「パフォーマンス」「品質」は独立したチェックなので並列実行できます。「差分を読む」は全ての前提なので最初に。「レビューまとめ」は全ての結果が必要なので最後に。この依存関係の整理がPhase分割の核心です。

### Step 3: ゲートを設定する

「人間の判断が必要な箇所はどこか？」を考えます。

ゲートを置くべき箇所の判断基準は以下の通りです。

- AIの判断だけでは不十分な箇所（ビジネス判断が必要）
- 間違った方向に進むと手戻りが大きい箇所
- ユーザーの確認なしに進めるとリスクがある箇所

コードレビューの例では以下のようになります。

```
ゲート1: Phase 2 完了時
  → セキュリティリスクが見つかった場合、修正方針はユーザーに確認
  → 理由: セキュリティ対策の方針はビジネス判断を含むため

ゲート2: Phase 6 完了時
  → レビューコメントの最終確認
  → 理由: AIの指摘が的外れな場合、そのまま投稿すると問題になるため
```

ゲートが多すぎる場合は減らしていいです。すべてのPhaseにゲートを入れると、AIに任せている意味がなくなります。「自動判断できる箇所」と「人間が判断すべき箇所」を区別することが大切です。

### Step 4: 各Phaseの実行手順を定義する

各Phaseで「AIが何をするか」を具体的に書きます。

3つの要素を意識します。

- **入力**: 何を読むか（どのファイル、どのデータ）
- **処理**: 何をするか（どんな分析、どんなチェック）
- **出力**: 何を返すか（どんなフォーマットで）

```markdown
### Phase 2: セキュリティチェック

**入力**: Phase 1 で特定した変更ファイル一覧

**処理**:
1. 各ファイルについて、OWASPトップ10の観点でチェックする
   - A01: アクセス制御の不備
   - A02: 暗号化の失敗
   - A03: インジェクション
   - A07: 認証の不備
2. SQLクエリの組み立てにユーザー入力が直接使われていないか確認する
3. 認証・認可のチェックが適切に行われているか確認する

**出力**:
以下の形式でレポートを作成する

リスクレベル: 高/中/低
対象ファイル: ファイルパス
問題の概要: 1行で説明
推奨対策: 具体的な修正案
```

出力フォーマットを定義しておくと、AIの出力が安定します。定義しないと、AIは毎回違うフォーマットで出力します。

### Step 5: エラーケースを定義する

「AIが困る場面」を想像して、その対処方法を書きます。

```markdown
## エラー時の振る舞い

- 対象ファイルが見つからない場合:
  → エラーメッセージを表示し、ユーザーにパスを確認する
  → 推測でパスを補完しない

- PRの差分が大きすぎる場合（変更ファイル50以上）:
  → ユーザーにスコープの絞り込みを提案する
  → 「全ファイルをレビューしますか？ それとも特定ディレクトリに絞りますか？」

- セキュリティリスクの判断に自信がない場合:
  → 確信度を明示する（「この箇所はセキュリティリスクの可能性があります（確信度: 60%）」）
  → 確信度が低い場合はユーザーに確認を求める

- レビュー中にテストが実行できない場合:
  → テスト実行をスキップし、その旨をレポートに明記する
  → 「テスト環境の問題でカバレッジの確認はスキップしました」
```

エラーケースを書かないと、AIは「行き詰まったら黙って止まる」か「独自の判断で続行する」かのどちらかになります。どちらも望ましくありません。

### Step 6: commands/ にエントリポイントを作る

ここまでで skills/ の内容は完成です。あとは commands/ にエントリポイントを作るだけです。

```markdown
---
description: "コードレビューを実行する"
---

$ARGUMENTS を対象にスキル `~/.claude/skills/review/review.md` を実行してください。

## Arguments
- 第1引数: レビュー対象（PRのURL、ファイルパス、ブランチ名）
- `--security-only`: セキュリティチェックのみ実行
- `--quick`: Phase 2-4 を並列実行し、ゲートを最終確認のみにする
```

コマンドファイルはこれだけです。ロジックはすべて skills/ に書いてあるので、コマンドファイルは「どのスキルを使うか」と「どんな引数を受け取るか」を書くだけで済みます。

### Step 7: テストして改善する

実際にコマンドを実行して、期待通りに動くか確認します。

```
/review https://github.com/example/repo/pull/123
```

確認するポイントは以下の通りです。

- Phase が定義した順番で実行されるか
- ゲートで実際に止まるか（止まらずに先に進んでしまわないか）
- 出力フォーマットが定義通りか
- エラーケースで正しく振る舞うか

最初の実行で完璧に動くことはほとんどありません。私のSFADも、最初のバージョンから20回以上修正しています。

よくある修正ポイントは以下の通りです。

- Phaseの粒度が粗すぎる → より細かく分割する
- Phaseの粒度が細かすぎる → まとめる
- ゲートが多すぎる → 自動判断できる箇所はゲートを外す
- 出力が期待と違う → 出力フォーマットをより具体的に定義する
- AIが手順を飛ばす → 「必ず」「この順番で」など、制約を強調する言葉を追加する

## 3つの実践例

SFADの他に、Skillとして実装できるワークフローの例を2つ示します。

### 例1: コードレビューSkill

Step 1-7 で作った例をそのまま使います。ファイル構成は以下の通りです。

```
~/.claude/skills/review/
  ├── SKILL.md          ← レビュー基準とサブスキル一覧
  └── review.md         ← Phase 1-6 の詳細手順
~/.claude/commands/review/
  └── review.md         ← /review コマンドのエントリポイント
```

SKILL.md の内容:

```markdown
---
name: review
description: "コードレビュー - セキュリティ、パフォーマンス、品質、テストカバレッジを体系的にチェック"
---

# Code Review Skill

## Sub-Skills
- /review review — フルレビュー実行

## Principles
1. セキュリティリスクは最優先で報告する
2. 推測ではなくコードの証拠に基づいて指摘する
3. 問題の指摘だけでなく、具体的な修正案を提示する
4. 重要度の高い指摘から順に報告する
```

review.md のPhase構成:

```markdown
# Code Review — 体系的コードレビュー

## Phase 1: PR差分の読み込みとスコープ特定
1. 変更されたファイルの一覧を取得する
2. 変更行数を集計する
3. 影響を受けるモジュールを特定する

## Phase 2: セキュリティチェック（OWASPトップ10）
1. SQLインジェクション: ユーザー入力がクエリに直接使われていないか
2. XSS: ユーザー入力がHTMLに直接埋め込まれていないか
3. 認証の不備: 認証チェックが適切に行われているか
4. アクセス制御: 権限チェックが漏れていないか
出力: リスクレベル（高/中/低）、対象ファイル、問題の概要、推奨対策

### ゲート: セキュリティリスク発見時
高リスクの問題が見つかった場合、ユーザーに修正方針を確認する。
低・中リスクの場合はレポートに記載して続行する。

## Phase 3: パフォーマンス分析
1. N+1クエリ: ループ内でDBアクセスしていないか
2. メモリリーク: 不要なオブジェクト保持がないか
3. 不要な再計算: キャッシュ可能な処理が毎回実行されていないか
出力: パフォーマンスリスク一覧

## Phase 4: コード品質チェック
1. SOLID原則の違反がないか
2. 命名が意図を表現しているか
3. 重複コードがないか
4. 関数・メソッドの長さが適切か（30行を超えていないか）
出力: 品質改善提案一覧

## Phase 5: テストカバレッジ確認
1. 変更されたコードに対するテストが存在するか
2. ハッピーパスと異常系の両方がテストされているか
3. 境界値テストがあるか
出力: テスト充足度の評価

## Phase 6: レビューまとめ
1. Phase 2-5 の結果を統合する
2. 重要度順にソートする（セキュリティ > パフォーマンス > 品質 > テスト）
3. 各指摘に具体的な修正案を付ける

### ゲート: 最終確認
レビューコメントの全文をユーザーに提示し、投稿の承認を得る。

## エラー時の振る舞い
- ファイルが見つからない: ユーザーにパスを確認する
- PRが大きすぎる（50ファイル以上）: スコープの絞り込みを提案する
```

### 例2: ドキュメント生成Skill

APIドキュメントを自動生成するSkillの例です。

```
~/.claude/skills/docs/
  ├── SKILL.md          ← ドキュメント標準と品質基準
  └── api-docs.md       ← APIドキュメント生成の詳細手順
~/.claude/commands/docs/
  └── api-docs.md       ← /docs api-docs コマンド
```

api-docs.md のPhase構成:

```markdown
# API Documentation Generator

## Phase 1: エンドポイントのスキャン
1. 指定されたディレクトリからルーティング定義を読み取る
2. 各エンドポイントのHTTPメソッド、パス、ハンドラを一覧化する
3. 既存のドキュメントがあれば差分を特定する

## Phase 2: リクエスト/レスポンスの型解析
1. 各エンドポイントのリクエストボディの型を抽出する
2. レスポンスボディの型を抽出する
3. ステータスコードごとのレスポンス形式を特定する

## Phase 3: ドキュメント生成
1. 以下の形式で各エンドポイントのドキュメントを生成する:
   - エンドポイント: メソッド + パス
   - 説明: 1行の概要
   - リクエストパラメータ: 名前、型、必須/任意、説明
   - リクエストボディ: JSON形式の例
   - レスポンス: ステータスコード別のJSON形式の例
   - エラーレスポンス: エラーコード別の説明

### ゲート: ドキュメント確認
生成されたドキュメントをユーザーに提示し、内容の確認を得る。

## Phase 4: ファイル出力
1. 確認されたドキュメントを指定形式で出力する（Markdown / OpenAPI）
2. 既存ドキュメントがある場合は差分のみ更新する

## エラー時の振る舞い
- ルーティング定義が見つからない: フレームワークの種類をユーザーに確認する
- 型情報が不完全: 不明な型は「unknown」と明記し、ユーザーに確認する
```

commands/api-docs.md:

```markdown
---
description: "APIドキュメントを生成する"
---

$ARGUMENTS を対象にスキル `~/.claude/skills/docs/api-docs.md` を実行してください。

## Arguments
- 第1引数: 対象ディレクトリ（例: @src/routes/）
- `--format markdown`: Markdown形式で出力（デフォルト）
- `--format openapi`: OpenAPI形式で出力
- `--update`: 既存ドキュメントの差分更新
```

### 例3: デプロイメントチェックリストSkill

デプロイ前の確認作業をSkill化する例です。

```
~/.claude/skills/deploy/
  ├── SKILL.md          ← デプロイメント基準
  └── checklist.md      ← デプロイチェックリストの手順
~/.claude/commands/deploy/
  └── checklist.md      ← /deploy checklist コマンド
```

checklist.md のPhase構成:

```markdown
# Deployment Checklist

## Phase 1: 環境確認
1. デプロイ先の環境を特定する（staging / production）
2. 環境変数の設定状況を確認する
3. 依存サービスの稼働状況を確認する

### ゲート: 環境確認
production環境へのデプロイの場合、ユーザーに明示的な確認を取る。
「production環境にデプロイします。続行しますか？」

## Phase 2: テスト実行
1. ユニットテストを全件実行する
2. インテグレーションテストを実行する
3. テスト結果を報告する
4. カバレッジが基準値（80%）を下回る場合は警告する

### ゲート: テスト結果確認
テストが1件でも失敗している場合、デプロイを中断する。
ユーザーに失敗したテストの一覧を提示し、続行するか判断を仰ぐ。

## Phase 3: ビルド
1. プロダクションビルドを実行する
2. ビルドエラーがないことを確認する
3. バンドルサイズを報告する

## Phase 4: デプロイ実行
1. デプロイコマンドを実行する
2. デプロイログを監視する
3. エラーが発生した場合は即座に報告する

## Phase 5: ヘルスチェック
1. デプロイ後にヘルスチェックエンドポイントを呼び出す
2. レスポンスが200であることを確認する
3. 主要機能のスモークテストを実行する

### ゲート: ヘルスチェック結果
ヘルスチェックが失敗した場合、ロールバックを提案する。

## Phase 6: ロールバック手順確認
1. ロールバックコマンドを表示する
2. ロールバックの判断基準を提示する
3. 監視期間（30分）を設定する

## エラー時の振る舞い
- ビルドが失敗: デプロイを中断し、エラーの詳細を報告する
- デプロイ中にエラー: 即座に停止し、ロールバックを提案する
- ヘルスチェック失敗: ロールバック手順を表示し、ユーザーの判断を仰ぐ
```

## よくある間違いと対策

Skillを作り始めた頃に私がやった間違いと、その対策をまとめます。

### 間違い1: Skillが長すぎる

1つのファイルに全ての手順を詰め込むと、AIが混乱します。Phase 10まであるSkillを作ったことがありましたが、後半のPhaseでAIが前半の制約を忘れることがありました。

**対策**: サブスキルに分割します。SFADではcycle.md（全体サイクル）を、spec.md/test.md/impl.md/reverse.md（個別手順）に分割しました。各ファイルは独立して実行できます。目安として、1ファイル300行を超えたら分割を検討します。

### 間違い2: ゲートが多すぎる

全てのPhaseにゲートを入れると、AIに任せている意味がなくなります。最初のバージョンでは8つのPhaseに5つのゲートを入れましたが、「毎回止まるので面倒」というフィードバックを受けました。

**対策**: ゲートは「AIの判断だけでは不十分な箇所」だけに入れます。「テストが全てGreenか」はAIが自動判定できるのでゲートにしません。「仕様が正しいか」は人間しか判断できないのでゲートにします。

### 間違い3: 抽象的すぎる

「コードを分析してください」「良い感じにレビューしてください」という記述では、AIは期待と違う動きをします。

**対策**: 具体的な出力フォーマットを定義します。「問題を報告する」ではなく「リスクレベル（高/中/低）、対象ファイル、問題の概要、推奨対策の形式で報告する」と書きます。入力・処理・出力を明確にすることで、AIの動作が安定します。

### 間違い4: エラーケースを忘れる

ハッピーパスだけ定義して、エラーケースを定義しないと、AIが行き詰まったときの挙動が予測できません。

**対策**: 「AIが困る場面」を想像します。「ファイルが見つからなかったら？」「テストが通らなかったら？」「判断に自信がなかったら？」。少なくとも3つのエラーケースは定義しておくことをお勧めします。

### 間違い5: 言語固有にしすぎる

「pytestでテストを実行する」と書くと、TypeScriptプロジェクトでは使えません。

**対策**: Auto-Detection + Stack Matrixで汎用化します。SFADでは「テストを実行する」という抽象的な手順を定義し、Auto-Detectionセクションで「Pythonならpytest、TypeScriptならvitest、Goならgo test」という対応を示しています。

## 良いSkillと悪いSkillの違い

最後に、良いSkillと悪いSkillの特徴を比較します。

### 良いSkillの特徴

- **Phase分割が明確**: 各Phaseが独立した工程として完結している
- **ゲートが適切**: AIの判断だけでは不十分な箇所にだけ配置されている
- **エラー処理がある**: AIが行き詰まったときの行動が定義されている
- **出力フォーマットが定義されている**: AIの出力が安定する
- **理論的基盤がある**: AIが「なぜこの手順なのか」を理解できる
- **Auto-Detectionがある**: 複数の言語やフレームワークに対応できる

### 悪いSkillの特徴

- **1つの巨大ファイル**: 全ての手順が1ファイルに詰め込まれている
- **ゲートなし**: AIが最初から最後まで止まらずに走る（暴走リスク）
- **「うまくやって」的な曖昧指示**: AIの解釈に依存する不安定な動作
- **エラー処理なし**: AIが行き詰まったとき黙って止まるか延々とリトライする
- **出力フォーマット未定義**: 毎回違う形式で出力される
- **理論的基盤なし**: AIが「なぜこうするのか」をユーザーに説明できない

良いSkillと悪いSkillの違いは、結局「どれだけ具体的に定義したか」に帰着します。AIは書かれた通りに動きます。書かれていないことは推測するか無視します。推測が当たる保証はありません。

## Skillを作ることの価値

Skillを作る作業は、実は「自分の暗黙知を言語化する作業」です。

「私がコードレビューするとき、何を見ているのか？」という問いに答えようとすると、普段は無意識にやっていることを意識的に整理する必要があります。「なんとなくセキュリティをチェックしている」を「OWASPトップ10のうちA01, A02, A03, A07を確認している」と具体化する作業です。

この言語化のプロセス自体に価値があります。Skillを作り終わったとき、あなたは自分のワークフローを以前より深く理解しているはずです。

そして言語化された知識は、Skillとして実装すれば再現可能になります。あなたが休んでいるときも、チームの新メンバーも、同じ品質でレビューやデプロイができるようになります。

「属人的なノウハウ」が「チームの資産」に変わる瞬間です。
