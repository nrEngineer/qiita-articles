## はじめに

この記事は、AI 駆動開発が広まる中で **「実装者の理解度」** という古典的だけど避けて通れないテーマを扱います。AI ツールやプロセスをいくら整えても、実装者がその機能を **頭から終わりまで説明できない** 状態だと、どんな PR を出してもどこかで崩れます。

具体的な事例（匿名化済み）と、シニア / テックリードがジュニアをコーチングするときの実践的な手順、そして「ブラウザで動いた」を検証完了と勘違いするパターンの矯正法を書きます。

有料記事にした理由は、現場の温度感を伴うコーチング論で、無料で広く拡散させるよりも **本気で読みたい人にだけ届けたい** からです。

---

## TL;DR

- ある PR で +3278 行の機能追加が上がってきたが、**認可ゼロ・冪等性破綻・型変換バグ**だらけだった
- 実装者の検証手段は **「ブラウザで動かす」** 一択。これでは致命的問題を 1 個も検出できない
- 根本原因は実装者が **そのフローを最初から最後まで自分の言葉で説明できなかった** こと
- AI に書かせるのは構わないが、**書かせた結果を本人がレビューできる頭** が無いと事故る
- コーチングの実践: 5 つの矯正ステップを順番にやってもらう

## ある日の Slack やり取り

匿名化のため詳細は変えていますが、こういう会話がありました。

> **シニア (テックリード):**
> @ジュニア
> これセキュリティチェックで絶対つつかれると思うので自分でもセルフレビューしてみてください
> 新しい API 全部に認可が無いです
> 誰でも API 叩ける状態になっているんですが問題ないんです？
>
> あとロジックで気になったのが何件か
> 順位計算の処理で小数切り捨てしてるのは問題ないですかね？
>
> AIに書かせたあと、AIにセルフレビューってさせてますか？
>
> **ジュニア:**
> @シニア
> みてくださってありがとうございます、確認します・・・
> 最後にテストはさせたのですが、セルフレビューという意味だとさせてないかもです・・・
> シニアさんはどのようにいつもやらせていますか？
> セルフレビューして　っていえばいいという話ですかね・・？
>
> **シニア:**
> 書かせたあとに振り返りは絶対したほうがいいです
> 自分が何を書かせたくてAIに指示を出したのか
> その結果がちゃんと満たされているのか
>
> .DS_Store ってなんでしたっけあと…
> pylint も結構 .pre-commit-config.yaml に disable でルール自体無効化してるっぽく
> あんまりよくなさそう
>
> あと処理の共通化もされていないっぽい
> 同じ処理がいろんなところにありそうです

このやり取り、痛いほどリアルです。シニアが優しく聞いている形だけど、内容は致命的。「セキュリティチェックで絶対つつかれる」レベルの認可漏れがある PR をジュニアが上げて、「ブラウザで動いた」と思っている状態。

そして注目すべきはジュニアの返事:

> 「最後にテストはさせたのですが、セルフレビューという意味だとさせてないかもです・・・」

「テスト」と「セルフレビュー」を別物として認識している。**ここが核心です**。

---

## 「ブラウザで動いた = OK」の何がダメか

ジュニアの実装プロセスを再構築すると、こうなっていました。

```
1. AI に「マルチプレイヤー機能を実装して」と投げる
2. AI が動くコードを返す
3. ブラウザで動作確認 → 動いた → OK と判断
4. PR 上げる
```

この 4 ステップの間に **「自分が書こうとしたものと、出てきたものの差分を確認する」** が一切無い。「動いた = 正しい」になっている。

ブラウザで触って気付けるのは **ハッピーパスが動くこと** だけです。

ブラウザで触っても **絶対に気付かないもの**:

| 検出できない問題 | なぜブラウザでは見えないか |
|---|---|
| 認可の有無 | 自分が admin で叩いてるから当たり前に通る |
| 小数切り捨て | 自分のテストデータが整数だから |
| pylint disable | コードを見ないと分からない |
| `.DS_Store` | git status 見ないと分からない |
| 冪等性バグ | 再実行しないと分からない |
| N+1 / 性能 | 負荷かけないと分からない |
| セキュリティ | 攻撃しないと分からない |
| 個人情報リスク | スクリーンショットに何が映ってるか確認しないと分からない |
| 外部ライブラリ誤用 | ドキュメント読まないと分からない |
| 並行実行レース | 同時 N リクエスト送らないと分からない |

つまり、ジュニアの検証手段は **致命的問題を 1 個も検出できない手段** だったのです。これを本人に自覚させるのが第一歩。

## なぜジュニアは気付けないのか

「気付けないのは経験不足だから」で片付けると本質を見失います。経験不足の前に、もっと根本的な問題があります。

### 問題の本質: フローを脳内で持っていない

シニアが最初に質問すべきは「認可ある？」ではなくて、

> **「この機能、企画作成からプレイヤーが順位表を見るまで、どんな処理が、どの順番で、何を読み書きして起きるか、自分の言葉で 12 ステップで説明してみて」**

これです。説明できないなら、レビュー以前に **そのフローを理解していない** ということ。理解していないものを AI に書かせて、出てきたものを検証することは構造的に不可能。

ジュニアが PR を上げる時点で、頭の中にあるのは多分こんな感じです:

```
1. 機能を作って
2. 動くようにして
3. PR を出す
```

3 ステップ。詳細はぼんやり。これだと AI から出てきたコードに対して「これは何をしているか」を 1 行ずつ説明することが原理的にできません。

## コーチングの 5 ステップ

シニア / テックリードがジュニアにこの問題を矯正させるための実践手順です。順番が大事です。

### Step 1: フローを口頭で説明させる（コード見ずに）

PR を見せる前に、まず本人にフローを語らせます。

> 「コードを開かずに、自分の言葉で説明してください。
>  - 企画作成から順位表表示まで、最低 12 ステップで
>  - 各ステップで誰がトリガー、何のメソッド、何のテーブルを読み書き、何が外部に送られるか
>  - 関係するアクター（運営者・プレイヤー・自動ジョブ）を全部出して」

説明できないステップが出てきたら、そこが本人の理解の穴です。**この段階で「あ、自分はこのフローを理解してないんだ」と本人に気付かせる** のが最大のコーチング効果。

ここで「Slide とか書きながらでもいいですか？」と聞かれたら「いや、コード見ずに、口頭で」と返します。書きながらだと AI に書かせがちなので、純粋に頭の中の理解度を測ります。

### Step 2: テストでカバーしている範囲を答えさせる

次に、

> 「あなたが書いたテストは、今説明した 12 ステップのうち、どこからどこまでをカバーしていますか？テスト関数名で答えてください」

これでカバレッジの実感を持たせます。「順位表 API のテストはあるけど、認可テストはない」「ジョブの単体テストはあるけど、二重実行のテストはない」みたいに、**カバーしていない領域が言語化** されます。

ジュニアがよくやる勘違いは「テストが N 件ある = カバレッジが高い」。重要なのは **件数ではなく、何を守れているか**。

### Step 3: カバーしていない領域について「壊れたらどうなる」を聞く

> 「その『カバーしていない領域』について、もし本番で壊れたら誰がどう困りますか？」

例:
- 認可がないことが攻撃者にバレたら？ → ユーザーが他人のデータを書き換える → 全ユーザーから信頼喪失
- 採番ジョブが二重実行されたら？ → participants が 2 倍になる → 試合運営崩壊
- 外部 AI が落ちたら？ → 解析できない → 運営が手動で全部入力 → 締切オーバー

これを言語化させると、本人が「あ、これマジでヤバい」と腹落ちします。**他人事から自分事への変換** がここで起きます。

### Step 4: AI に敵対的レビューさせる手順を教える

ここで初めて AI レビューの話を出します。

> 「Claude / Cursor で **新しい会話** を開いて、こういうプロンプトを投げてみてください」

```
あなたはセキュリティとコード品質に厳しいシニアエンジニアです。
以下の PR を「絶対に通したくない」前提で、マージブロッカーを探してください。

特に以下を重点的に見てください:
- 認可・認証の漏れ
- SQL インジェクション、N+1
- 例外握りつぶし
- リソースリーク・トランザクション境界
- 個人情報の混入（ファイル、ログ、レスポンス）
- 設定ファイルの変更（.gitignore, pre-commit, lint disable）
- ロジックバグ（オフバイワン、型変換、丸め、冪等性）
- 既存機能への影響（リグレッション）

コードを貼ります： <git diff>
```

**ポイント**: 「**新しい会話で**」やること。同じ会話で「レビューして」と言うと AI は自分が書いたコードを擁護します。文脈リセット必須。これを教えないと、本人は同じ会話で AI に質問して「問題ありません！」と言われて満足します。

### Step 5: 修正を本人にやらせる（ここが学習の核心）

シニアがやってはいけないのは **「指摘を全部リストアップして本人に渡す」** こと。それやると次回も同じ事故を起こします。

代わりに:

> 「上のプロンプトで AI が出してきた指摘リストを、自分で優先順位付けして、修正案を考えてください。完了したら『何が壊れていて、何を直したか』を PR コメントに書いてください」

修正は本人が考える。シニアは「方針が正しいか」だけ確認する。

これを 3 PR 続けると、ジュニアは **「AI に書かせる前にフローを脳内で構築する」** クセが付きます。なぜなら、説明できない PR は Step 1 で必ず詰まるし、それが本人にとって苦痛だからです。苦痛を避けるために事前にフローを考えるようになる。これがコーチング。

## やってはいけないコーチング

実際にいくつかやらかしました。反省として書きます。

### NG 1: 修正案を全部書いて渡す

「ここを直して、ここをこう変えて、ここに認可つけて...」とリストにして渡すと、ジュニアは AI にそのリストを投げて修正させて「直しました！」で終わります。**本人の頭は動いていない**。次の PR で同じ事故を起こします。

### NG 2: 「セキュリティ的にダメ」とだけ言う

抽象的な指摘は本人に届きません。「未認証で curl で叩いたら 200 が返ってきます。試してみてください」と **具体的な再現手順** を渡す。本人に手を動かして再現させると、「あ、本当だ」が起きて初めて学びになります。

### NG 3: 「次から気をつけて」で終わる

これが最悪です。「気をつける」は実行可能なアクションではありません。具体的な手順（フロー説明・テスト範囲リスト・敵対的レビュー）を **習慣化させる** のが必要。

### NG 4: シニアが代わりに修正してマージ

時間的に追われていると、シニアが直してマージしたくなります。でもこれは長期的にはチームの破滅。ジュニアは「壊しても誰かが直してくれる」を学習します。

## 「AI を使う実装者」の学習サイクルを破壊する 3 つの罠

ジュニアが AI を使うと、伝統的な学習サイクルが破壊されます。

### 罠 1: 自分でハマる時間が無くなる

伝統的な学習は:

```
自分で書く → 動かない → 調べる → 理解する → 動く → レビュー → 直す → 学ぶ
```

AI を使うと:

```
AI が書く → 動く → PR → レビューで指摘 → AI に「直して」 → 動く → マージ
```

後者は **「自分が何を理解していないか」を理解しないまま進む**。指摘を受けても「AI に投げ直す」だけで本人の頭は動いていません。

### 罠 2: 「動いた」が達成感になる

AI が一発で動くコードを出してくる。それを本人が「動かした」気になる。実際は AI が書いたコードを本人がコピペしただけ。**達成感と実力の乖離** が起きます。

### 罠 3: コードが読めなくなる

自分で書かないと、コードを読む筋肉が衰えます。AI が出したコードを「動くから OK」で流すクセが付くと、他人の PR を読むときも「動くんだろ」で流すようになります。**レビュアーとしての成長が止まります**。

## 矯正の 3 つの実践

### 実践 1: 「自分の PR を 15 分で口頭説明する会」

週 1 で、ジュニアに自分の PR を 15 分で同期メンバーに口頭説明してもらいます。

ルール:
- スライド禁止
- AI 補助禁止
- コードを画面共有して指で追いながら説明
- 「ここ何やってますか？」をメンバーが質問していい

説明できなかったら「ここ理解してないね」と即指摘。これを 3 ヶ月続けると本人がコードを読むようになります。

### 実践 2: 「実装着手前に擬似コードを書く」

AI に書かせる前に `docs/wip/<feature>.md` に自分の言葉で:

- やること
- 関係するテーブル
- 関係する関数
- フロー（5 ステップ以上）
- 想定エラー

これを書いてから AI に渡す。**書けないなら理解していない**ので、まずシニアと相談する。AI 利用は「自分の設計を実装に翻訳する」道具であって「自分の代わりに考える」道具ではない、を体得させます。

### 実践 3: 「AI の出力を必ず疑う」

ジュニアに 1 週間、

> 「AI が書いたコードを必ず 5 個バグを見つけてから merge する」

というクオータを課します。バグが無さそうなら「もっと厳しく見ろ」と返す。最初は無理矢理探させる。

これで **「AI の出力には必ず問題がある」** という前提が体に入ります。今は **「AI 出力 = 完成品」** の前提なので、ここを反転させる必要があります。

## シニア側の心構え

最後にシニア / テックリード側に伝えたいこと。

### あなたが PR を 1 つずつ手で見ても、長期的にはスケールしない

シニアが致命的問題を 30 秒で検出できる能力を持っているのは素晴らしいけど、それを **毎回手で発動させる** のは持続不可能です。チームメンバーが増えると破綻します。

スケールさせるには **「シニアがやってる検出を仕組みに移植する」** のが正解。

- 認可テスト自動化 → シニアが「認可ある？」と聞かなくて済む
- bandit / pylint 強制 → シニアが「pylint disable 増えてる」と気付く前に CI が落ちる
- pre-commit で大ファイル禁止 → シニアが「.DS_Store 何？」と言う前に commit がブロックされる
- PR テンプレで「フロー 5 ステップ」必須 → シニアが「フロー説明できる？」と問わなくて済む

仕組み化に 1 週間投資する価値は、シニアの時間が空くこと以上に **ジュニアが事故を起こす前に CI で止まる** ことにあります。事故が止まると、ジュニアは「やらかして怒られる」を経験しなくて済むし、シニアは「怒る」を経験しなくて済みます。両方にとって良い。

### 1on1 で必ず聞くこと

ジュニアとの 1on1 で、こう聞いてください。

> 「最近 AI に書かせたコードで、自分が一番理解できていない部分はどこ？」

正直に答えてくれたら、それが学習機会です。「ない」と答えたら、それは黄信号です。誰でも理解できてない部分があるはずなので、「ない」と言うのは自己認識ができていない証拠。

## まとめ

- 実装者が **フローを 12 ステップで説明できない** なら、その PR は本人が検証できる範囲を超えている
- ブラウザで動いた = ハッピーパスが動いた、にすぎない。**致命的問題は 1 つも検出できない**
- コーチングの順番: フロー説明 → テストカバー範囲 → 壊れたら何が起きるか → 敵対的レビュー → 本人が修正
- やってはいけない: 修正案を全部書いて渡す、抽象的な指摘で済ます、シニアが代わりに直す
- 仕組み化（CI 認可テスト、pre-commit 大ファイル禁止、PR テンプレ必須項目）はシニアの時間を空け、ジュニアの事故を未然に止める
- 1on1 で「最近自分が一番理解できてないコードはどこ？」と必ず聞く

「AI 駆動開発時代のコーチング」って、結局のところ **昔ながらの設計力・読解力・批判的思考をどう学習させるか** という古典的なテーマなんだと思います。AI が書いてくれるからこそ、書く以外のスキル — 読む・疑う・分解する — が今まで以上に重要になっている。

そして、これを教えられるシニアがチームに 1 人でもいるかどうかが、長期的に AI 駆動開発がうまく回るかどうかの分水嶺になると思います。
