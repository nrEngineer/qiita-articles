## 仕様書がない引き継ぎの恐怖

ある業務システムの認証機能を引き継ぎました。コードはある。テストもある。でも「このコードが何をしているのか」「どういうルールで動いているのか」を説明できる人がいない状態でした。

認証機能はセキュリティの中心です。アカウントロック、OTP（ワンタイムパスワード）、パスワードポリシー、トークン管理。どれも「なぜこの実装になっているのか」が重要な領域です。

コードは「何をしているか」を教えてくれます。でも「何をすべきか」は教えてくれません。

ここに根本的な問題があります。コードを読めば `failed_count >= 5` でロックすることはわかります。でも「5回でロックするのが正しいのか」はわかりません。3回にすべきなのか、10回でいいのか、それとも5回に何か根拠があるのか。コードからは読み取れません。

これは「バグか仕様かわからない」問題です。挙動がおかしいと思っても、それが意図的な設計なのか、実装者のミスなのか、判断できません。

そしてこの問題は、Pythonに限った話ではありません。TypeScriptでも、Goでも、Javaでも、仕様書がないコードを引き継いだ瞬間に同じ恐怖が訪れます。

## 仕様考古学とは

この状況への対処法として、私は「仕様考古学」というアプローチを試しました。

考古学のメタファーで説明します。遺跡（コード）を発掘して、そこに暮らしていた文明（仕様）の記録を復元する作業です。

遺跡は残っています。建物の配置（関数の構造）、道具（変数）、記録の断片（コメントやテスト）から、当時の生活（意図された振る舞い）を推測します。

でも推測には限界があります。「なぜこの場所に神殿を建てたのか」は遺跡からは読み取れません。同様に、「なぜ5回でロックするのか」はコードからは読み取れません。

AIをアシスタント考古学者として使います。AIはコードを高速に読んで、パターンを検出し、振る舞いを構造化してくれます。ただし、「この発掘結果が正しいか」を判断するのは、人間の考古学者（私たち）の仕事です。

## reverseの7つのPhase

仕様考古学を体系的に行うために、7つのPhaseに分けて実行しました。この方法論自体は言語に依存しません。コードから振る舞いを抽出して構造化し、人間が承認するというプロセスは、どの言語のレガシーコードにも適用できます。

### Phase 1: 対象コードの特定

最初に、認証機能に関連するファイルを特定しました。

import文と依存関係を辿って、認証機能の境界を定めます。

```python
# こういうimport関係を追っていく
from app.auth.service import AuthService        # 認証サービス本体
from app.auth.models import User, LoginAttempt   # データモデル
from app.auth.otp import OTPManager              # OTP管理
from app.auth.tokens import TokenService         # トークン管理
from app.auth.password import PasswordPolicy     # パスワードポリシー
```

依存関係を辿ると、認証機能が以下のファイルにまたがっていることがわかりました。

| カテゴリ | ファイル数 | 内容 |
|---------|----------|------|
| サービス層 | 3 | 認証ロジック、OTP、トークン管理 |
| モデル層 | 2 | ユーザー、ログイン試行履歴 |
| ルーター層 | 1 | APIエンドポイント定義 |
| ユーティリティ | 2 | パスワードポリシー、暗号化 |
| テスト | 4 | 各レイヤーのテスト |

合計12ファイル。このスコープをAIに読ませます。

**言語を超えた教訓**: 対象コードの特定は、import/依存関係を辿るという手法でどの言語でも同じように行える。

### Phase 2: 振る舞い抽出

コードの条件分岐、エラー処理、型定義から「事実」を読み取ります。ここでは「意図」ではなく「事実」に集中します。

```python
# コードから読み取れる事実の例
if failed_count >= 5:
    lock_account(user_id)
    # 事実: failed_countが5以上のとき、アカウントがロックされる
    # 意図: 不明（5回の根拠がない）

otp = generate_otp(length=6, expiry_seconds=300)
    # 事実: OTPは6桁、有効期限300秒（5分）
    # 意図: おそらくセキュリティ要件だが、文書化されていない

if len(password) < 10:
    raise ValidationError("Password too short")
    # 事実: パスワードは10文字以上
    # 意図: パスワードポリシーの一部だが、なぜ10文字なのかは不明
```

AIに「このコードが何をしているか、事実だけを列挙して」と依頼します。この段階では「こうすべき」という判断はさせません。

**言語を超えた教訓**: 振る舞い抽出は、「事実」と「意図」を分離することが最も重要。AIは事実の抽出が得意だが、意図の推測は信頼できない。

### Phase 3: Example Map生成

Phase 2で抽出した事実を、BDD（振る舞い駆動開発）の構造に変換します。

Example Mapとは、Story（誰が何をしたいか）、Rule（ビジネスルール）、Example（具体例）、Question（未決事項）の4要素で仕様を整理する手法です。

```yaml
Story: ユーザーとして、安全にログインしたい

Rule: ログインにはメールアドレスとパスワードが必要
  Example: 正しいメールとパスワード → JWTトークンが発行される
  Example: 間違ったパスワード → エラーが返る、failed_countが1増える

Rule: 5回連続でログイン失敗するとアカウントがロックされる
  Example: 4回失敗 → まだログイン可能
  Example: 5回失敗 → アカウントロック、ログイン不可
  Question: 5回の根拠は何か？（OWASPガイドライン？社内規定？）
  Question: ロックの解除方法は？（時間経過？管理者操作？）

Rule: OTPは6桁で5分間有効
  Example: 正しいOTPを5分以内に入力 → 認証成功
  Example: 5分経過後に入力 → 期限切れエラー
  Question: OTPは常時有効だが、これは一時的な実装か？
```

事実をこの構造に並べると、「決まっていること」と「決まっていないこと」が可視化されます。

**言語を超えた教訓**: BDDのExample Map構造は、言語に依存しない仕様の記述フォーマットとして機能する。

### Phase 4: 不明点・矛盾の検出

Phase 3で生成したExample Mapから、6つの問題カテゴリでコードを分析します。

| カテゴリ | 意味 | 例 |
|---------|------|-----|
| SECURITY | セキュリティ上の問題 | レート制限なし、トークン露出 |
| DEAD CODE | 使われていないコード | 重複関数、no-opエンドポイント |
| INCONSISTENT | テストとコードの矛盾 | 引数の不一致 |
| MISSING | 存在すべき機能の欠落 | ブルートフォース対策なし |
| IMPLICIT | 暗黙の前提・マジックナンバー | ロック回数5回の根拠不明 |
| UNDOCUMENTED | 文書化されていないロジック | 計算式にコメントなし |

AIはコードとテストを突き合わせて、これらのカテゴリに該当する箇所を検出します。

**言語を超えた教訓**: 問題カテゴリの分類は言語に依存しない。どの言語のコードにも同じ6カテゴリが適用できる。

### Phase 5: ユーザーバリデーション

ここが最も重要なPhaseです。AIが出した結果を、人間が検証します。

AIは「コードがこう動いている」を整理するのは得意ですが、「この設計の意図は何か」「このトレードオフはなぜ選ばれたのか」までは判断できません。

このPhaseでは、以下を人間が決定します。

- AIが抽出したルールは正しいか
- Questionとして挙がった項目の答えは何か
- 問題として検出された項目のうち、実際に修正が必要なのはどれか
- 「IMPLICIT」タグがついた値（マジックナンバー等）には根拠があるのか

**言語を超えた教訓**: AIは材料を揃えるツールであり、「何が正解か」を決めるのは人間の仕事である。

### Phase 6: 仕様ファイル生成

人間が承認した結果を、正式な仕様ファイルとして出力します。

これにより、「仕様書がない」状態から「仕様書がある」状態に移行します。以降の開発やメンテナンスは、この仕様ファイルを基準に行えます。

### Phase 7: テスト生成

仕様ファイルに基づいて、現状の振る舞いを固定するテストを生成します。

このテストは「現状が正しい」ことを証明するものではありません。「現状の振る舞いを記録する」ものです。将来の変更で振る舞いが変わったとき、「意図的な変更なのか、デグレなのか」を判別するためのベースラインになります。

**言語を超えた教訓**: 既存コードのテストは「正しさの証明」ではなく「現状の記録」として書く。

## 確信度スコアリングの方法論

9つのRuleにつけた確信度は、以下の基準で判定しています。

**高（High Confidence）**: コードとテスト両方で確認できる

```python
# コード
MAX_LOGIN_ATTEMPTS = 5
if failed_count >= MAX_LOGIN_ATTEMPTS:
    lock_account(user_id)

# テスト
def test_account_locks_after_5_failures():
    for i in range(5):
        login(wrong_password)
    assert user.is_locked == True
```

コードの実装とテストの検証が一致しているので、「5回失敗でロックする」というルールは高確信度です。

**中（Medium Confidence）**: コードにあるがテスト不十分

```python
# コード
otp = generate_otp(length=6, expiry_seconds=300)

# テスト: OTP生成のテストはあるが、期限切れのテストがない
def test_generate_otp():
    otp = generate_otp(length=6, expiry_seconds=300)
    assert len(otp) == 6
    # expiry_seconds のテストが存在しない
```

OTPが6桁であることはテストで確認されていますが、「5分で期限切れになる」ことはテストされていません。コードにはあるけれど、検証が不十分です。

**低（Low Confidence）**: 意図不明、矛盾あり

コードに実装はあるが、テストと矛盾している、または意図が読み取れない場合です。

確信度が低いRuleこそ、ユーザーに確認すべき箇所です。高確信度のRuleはコードとテストが整合しているので、そのまま仕様として採用できます。低確信度のRuleは「本当にこれで正しいのか？」を関係者に確認する必要があります。

**言語を超えた教訓**: 確信度スコアリングにより、「何を信頼してよいか」と「何を人間に確認すべきか」が明確になる。

## 特に重要な問題3件の詳細

### 問題1: `ast.literal_eval()` のセキュリティリスク

11の問題の中で、最も驚いたのがこれでした。

```python
# 問題のあったコード
def parse_token(token_str: str):
    return ast.literal_eval(token_str)  # 外部入力をそのまま評価
```

`ast.literal_eval()` はPythonのリテラル（文字列、数値、リスト、辞書など）を評価する関数です。`eval()` よりは安全とされていますが、外部から受け取った文字列をそのまま渡すのは危険です。

なぜこのコードが書かれたのか。おそらく、開発初期にJSONではない形式でトークンを保存していた名残です。フォーマットの移行時に `json.loads()` に差し替えるべきだったのが、そのまま残ってしまったのでしょう。

修正は単純です。

```python
# あるべき姿
import json

def parse_token(token_str: str) -> dict:
    try:
        return json.loads(token_str)
    except json.JSONDecodeError as e:
        logger.error(f"Token parse failed: {e}")
        raise ValueError(f"Invalid token format: {e}") from e
```

この問題は「AIが見つけた」のではなく、「AIがコードを読む過程で浮かび上がった」ものです。人間がコードレビューで見落としていた箇所を、AIが振る舞い抽出の一環として検出しました。

**言語を超えた教訓**: 外部入力の評価は、どの言語でも最大級のセキュリティリスクである。JavaScriptの `eval()`、Goの `reflect` の不適切な使用、Rustの `unsafe` ブロックなど、同じパターンはあらゆる言語に存在する。

### 問題2: パスワードリセットのレート制限なし

```python
# 問題のあったコード
@router.post("/password-reset")
async def request_password_reset(email: str):
    user = await get_user_by_email(email)
    if user:
        await send_reset_email(user)
    return {"message": "If the email exists, a reset link has been sent"}
```

このエンドポイントには、リクエスト回数の制限がありませんでした。攻撃者が大量のリクエストを送ると、以下のリスクがあります。

- **メールボム攻撃**: 特定のメールアドレスに大量のリセットメールを送りつける
- **アカウント列挙攻撃**: レスポンスタイムの差異から、存在するアカウントを特定する
- **リソース枯渇**: メール送信サービスのクォータを使い切る

レスポンスのメッセージは適切です。「メールが存在する場合はリセットリンクを送信しました」という表現は、アカウントの存在を漏らさない設計です。しかし、レート制限がなければ根本的な防御になりません。

```python
# あるべき姿
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

@router.post("/password-reset")
@limiter.limit("3/minute")
async def request_password_reset(request: Request, email: str):
    user = await get_user_by_email(email)
    if user:
        await send_reset_email(user)
    return {"message": "If the email exists, a reset link has been sent"}
```

レート制限の概念は言語に依存しません。TypeScriptなら `express-rate-limit`、Goなら `golang.org/x/time/rate`、Rustなら `tower::limit` など、どの言語にも対応するライブラリがあります。重要なのは「レート制限が必要であるという認識」です。

**言語を超えた教訓**: レート制限は認証系エンドポイントの最低限の防御である。言語やフレームワークに関係なく、認証関連のエンドポイントにはレート制限が必須。

### 問題3: IMPLICIT ― マジックナンバーの発見

```python
# コードに書いてあった定数
MAX_LOGIN_ATTEMPTS = 5  # ← なぜ5回なのか？
```

この「5」に根拠が示されていませんでした。定数として切り出されているだけましですが、「なぜ5なのか」は誰にもわかりません。

マジックナンバーの問題は、値そのものではありません。「変えてよいのかどうかわからない」ことが問題です。

- 3回にしたらユーザーが頻繁にロックアウトされる
- 10回にしたら総当たり攻撃のリスクが上がる
- 5回にはOWASPの推奨や社内のセキュリティ規定に基づく根拠があるかもしれない

コードにもドキュメントにも書かれていなければ、後続の開発者は「触らない」を選びます。結果として、要件が変わっても修正されない死んだ定数になります。

```python
# あるべき姿
# OWASP推奨値に基づく。変更時はセキュリティレビュー必須。
# ref: https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html
MAX_LOGIN_ATTEMPTS = 5
```

コメント一行で、後続の開発者が「根拠がある」「変更手順がある」と判断できるようになります。

この問題はPython特有ではありません。TypeScriptの `const MAX_RETRIES = 3`、Goの `maxRetries := 3`、Rustの `const MAX_RETRIES: u32 = 3` ── どの言語でも、根拠のないマジックナンバーは同じ問題を引き起こします。

**言語を超えた教訓**: マジックナンバーの本当の問題は「なぜその値なのか」がわからないことであり、これはどの言語でも同じ構造を持つ。

## 言語を超えた抽象原則

ここまでの内容を、言語に依存しない原則として整理します。

### 「仕様書がないコード」はどの言語にも存在する

TypeScriptのReactアプリケーション、GoのマイクロサービスAPI、RustのCLIツール ── 仕様書がないまま引き継がれるコードは、言語やフレームワークを問わず存在します。

担当者が退職した、ドキュメントが更新されなかった、そもそもドキュメントを書く文化がなかった。理由はさまざまですが、結果として「コードだけが残っている」という状況は同じです。

### reverseの方法論は言語に依存しない

7つのPhaseの本質を言語に依存しない形で整理します。

| Phase | 本質 | 言語固有の要素 |
|-------|------|--------------|
| 1. 対象コードの特定 | 依存関係の追跡 | import/require/use構文の違いのみ |
| 2. 振る舞い抽出 | 条件分岐とエラー処理からの事実読み取り | if/match/switch構文の違いのみ |
| 3. Example Map生成 | 事実のBDD構造への変換 | 言語に依存しない |
| 4. 不明点・矛盾の検出 | 6カテゴリでの分類 | 言語に依存しない |
| 5. ユーザーバリデーション | 人間による承認 | 言語に依存しない |
| 6. 仕様ファイル生成 | 承認結果の文書化 | 言語に依存しない |
| 7. テスト生成 | 現状の振る舞い固定 | テストフレームワークの違いのみ |

Phase 1と2は言語固有の構文に依存しますが、Phase 3以降は完全に言語非依存です。つまり、一度この方法論を身につければ、どの言語のレガシーコードにも適用できます。

### AIの役割と限界

AIは以下のことが得意です。

- コードを高速に読んで構造を把握する
- 条件分岐やエラー処理からルールを抽出する
- テストとコードの矛盾を検出する
- 問題をカテゴリに分類して深刻度を判定する

AIが苦手なこと（人間が判断すべきこと）は以下です。

- 「なぜこの設計が選ばれたのか」の推測
- ビジネス要件との整合性の判断
- 「バグか仕様か」の最終判定
- セキュリティ上の問題の修正優先度の決定

## 人間が判断すべきポイント

AIの出力は「材料」であって「答え」ではありません。

今回の分析でも、AIが間違えた箇所がありました。

- OTPの実装を「常時有効」と判定したが、実際はフラグで切り替えられる設計だった。コード上はフラグが常に `True` に設定されていただけで、設計としては切り替え可能だった。
- 「重複関数」とフラグされた関数のうち1件は、意図的に別の振る舞いを持たせていた。関数名とシグネチャが似ていたため、AIは重複と判定した。

AIは「コードから読み取れる事実」を整理するのは得意ですが、「コードに書かれていない意図」を読み取ることはできません。

人間が判断すべきポイントをまとめます。

- **確信度「中」のRule**: 仕様が曖昧なのか、実装が曖昧なのかを確認する。仕様が曖昧ならステークホルダーに確認が必要。実装が曖昧ならリファクタリングの対象。
- **Questionリスト**: AIが「わからない」と言った箇所は、設計上の決定事項を確認する場所。答えが見つかったら仕様に明記する。
- **[IMPLICIT]タグの問題**: 根拠があって意図的にその値を選んでいる可能性がある。変更する前に確認する。
- **[DEAD CODE]タグの問題**: git blameで最後の変更者と理由を確認する。「使われていない」ことと「不要である」ことは別。

## まとめ

仕様書がない認証機能のコードをAIに読ませた結果、以下が得られました。

- **9つのRule**: 確信度付きで、コードの振る舞いを構造化できた
- **4つのQuestion**: AIが「わからない」と言った箇所が、実際に仕様が曖昧な箇所と一致した
- **11の問題**: セキュリティ上のCriticalが3件を含む問題を検出できた

仕様考古学は「コードから仕様を復元する」手法です。完璧な仕様書は得られませんが、「何がわかっていて、何がわかっていないのか」を明確にできます。

この手法は言語に依存しません。コードから振る舞いを抽出し、構造化し、人間が承認する。このプロセスはPythonでもTypeScriptでもGoでもRustでも同じです。

重要なのは、AIは材料を揃えるツールであるということです。「何が正解か」を決めるのは人間です。AIの出力を鵜呑みにせず、確信度スコアとQuestionリストを使って「どこを人間が確認すべきか」を判断してください。
