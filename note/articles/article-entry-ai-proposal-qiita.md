「AIツール入れたいんですが」

「品質下がらない？」

「誰が責任取るの？」

「予算は？」

この3連続パンチで、提案書がゴミ箱に直行した経験はないでしょうか。

私はあります。GitHub Copilotを試して「めちゃくちゃ効率上がりました！」と上司に話しに行った日のことを、今でも覚えています。返ってきたのは「感想はわかった。で、品質はどう担保するの？」という一言でした。

答えられませんでした。

この記事は、そのときの私に向けて書いています。

## なぜAI導入提案は通らないのか

通らない提案には、共通の構造があります。

### 理由1: 感想で語っている

「便利でした」「効率が上がります」「コードが速く書けます」

上司や経営層がこれを聞いても、意思決定の根拠にはなりません。「便利」の定義は人によって違うし、「効率」の数字がなければ投資対効果の判断ができない。

主観的な体験談は、共感は得られても稟議は通りません。

### 理由2: リスク対策がない

「AIが変なコード書いたらどうするの？」

この質問に「大丈夫です」と答えるのは、最も危険な回答です。上司は「この人はリスクを把握していない」と判断します。

リスクがないことを主張するより、「このリスクは、この仕組みで対処します」と言える状態にする必要があります。

### 理由3: 数字がない

「どれくらい効果があるの？」「投資回収できるの？」

ROIを示せないと、優先度の判断ができません。予算を確保する根拠がないと、決裁者は動けません。

感想を語る人は多い。数字とリスク対策を持ってくる人は、ほとんどいない。この差が、提案の通過率を決定します。

## 武器1: データで語る ― 78バグ分析の示すこと

ある業務システムをClaude Codeで開発し、リリース後に78件のバグを記録しました。全件を分析した結果、**59%（46件）は品質基盤（lint・CI・テスト）の導入だけで防止可能でした**。データの完全版はArt.1「AIと開発して78個のバグを踏んだので全部分類した」で公開しています。

重要な問いは「このバグは、AIが原因か？」ではなく「このバグは、防げたか？」です。

「AIが書いたから品質が下がる」のではありません。「品質基盤のない状態でAIを使っていた」のが原因です。

この2つは、根本的に違います。

前者なら「AIを入れない」が正しい結論になります。後者なら「品質基盤を先に整備してからAIを導入する」が正しい結論です。そして後者の方が、証拠に基づいています。

上司への説明に使えるフレームは、これです。

> 「AIが品質を下げるのではなく、品質基盤がないことが問題でした。品質基盤を先に整備することで、AIによる開発速度の向上と品質の維持を両立できます。」

## 武器2: ROIで語る ― 662%の計算根拠

ROIを持って提案に行くと、会話が変わります。

### 投資コスト（Day 0）

初期整備に必要な工数です。

| 作業 | 工数 |
|---|---|
| lint strict設定（Python/TypeScript/Go） | 2時間 |
| CI/CD パイプライン設定 | 1時間 |
| テスト基盤構築（pytest/Vitest/testing） | 1時間 |
| **合計** | **4時間** |

エンジニア単価を5,000円/時間とすると、**投資額は2万円**です。

### 回収効果（最初の1ヶ月）

lint + 型チェックで防止できる44件のバグを対象に計算します。

| 項目 | 数値 | 根拠 |
|---|---|---|
| 防止可能なバグ数 | 44件 | lint + 型チェック + CI による防止 |
| バグ1件あたりの平均修正コスト | 2,500円 | 発見0.5h × 5,000円/h + 修正0.5h × 5,000円/h |
| 合計回収額 | **110,000円** | 44件 × 2,500円 |

**ROI = (110,000 - 20,000) ÷ 20,000 × 100 = 450%**

さらに、バグ修正コストは「本番に出てから発見された場合」に跳ね上がります。本番バグの修正コストは開発中バグの4〜10倍と言われています（IBM研究・NIST報告など）。保守的に4倍で計算すると、修正コストは1件あたり1万円。

| 項目 | 数値 |
|---|---|
| 防止可能なバグ数（本番想定） | 23件（lint で防げる print残留 + bare except） |
| 本番バグ1件あたりの修正コスト | 10,000円 |
| 回収額 | 230,000円 |
| 投資額 | 20,000円 |
| **ROI** | **1,050%** |

平均的な試算で**662%のROI**を見込めます（本番バグ想定の中間値での計算）。

補足: 450%（開発中バグのみ）〜1,050%（本番バグ想定）の範囲で、本番混入率50%を想定した中間値が662%です。自社の本番バグ比率に合わせて調整してください。

```

回収額 = 防止バグ数[件] × 平均修正コスト[円/件]
ROI   = (回収額 - 投資額) ÷ 投資額 × 100 [%]
```

### 月次の効果測定指標

導入後に「効果が出ているか」を数字で追うための指標です。

| KPI | 計測方法 | 目標値 |
|---|---|---|
| バグ密度（/KLOC） | バグ件数 ÷ コード行数(1,000行単位) | 導入前比 -50% |
| CI成功率 | 成功ビルド数 ÷ 総ビルド数 | 85%以上 |
| 平均バグ修正時間 | バグ発見〜クローズまでの平均時間 | 導入前比 -30% |
| レビュー指摘数 | PRレビューでのコメント数 | 導入前比 -40% |

これらの数字を毎月報告できれば、「ROIが出ているか」の説明責任を果たせます。

## 武器3: リスク対策で語る ― ゲート設計という安全装置

「AIが暴走したらどうする？」

この質問への正しい答えは「暴走させない仕組みがあります」です。

SFAD（Spec-First AI Development）ワークフローでは、AIに「仕様を決める権限」を与えません。

### 3つのゲート

```
Phase 1: Discovery
  AIが情報収集・リスク洗い出し・不明点の列挙を行う
  ↓
  [ゲート1: Discovery承認] ← 人間が判断
  「AIが提示したリスクと不明点に合意するか」
  ↓
Phase 2: Spec
  AIが仕様案を作成する（Example Mappingで構造化）
  ↓
  [ゲート2: 仕様承認] ← 人間が判断
  「この仕様で実装して問題ないか」
  ↓
Phase 3: Implementation
  AIが仕様に基づいてコードを実装する
  ↓
  [ゲート3: エッジケース承認] ← 人間が判断
  「仕様に書いていなかった境界値の処理をこれでよいか」
  ↓
Phase 4: Validation
  自動テスト + 人間によるレビュー
```

AIが決定できるのは「実装の詳細」だけです。「何を作るか」「どう振る舞うか」は、すべてゲートで人間が承認します。

この設計思想を一言で表すと、「AIは材料を揃え、人間が判断する」です。

AIは無限に速く情報を集め、選択肢を整理し、コードを生成できます。でも「この判断が正しいか」の責任を取るのは人間です。ゲート設計は、この責任の所在を明確にする仕組みです。

上司への説明に使えるフレームは、これです。

> 「AIが仕様を決めることはありません。AIが提案→人間が承認という流れを、Discovery・仕様・エッジケースの3段階で設けています。承認なしにAIが勝手に進むことはありません。」

## 提案が通った後: 運用プランの骨格

稟議が通ってからも、計画が必要です。

### Month 0: 品質基盤整備（4時間）

```bash
# Python の場合
pip install pylint mypy black isort
# .pylintrc を strict設定で作成
# pre-commit フックで自動チェック

# TypeScript の場合
# tsconfig.json を strict: true に設定
# ESLint + @typescript-eslint/strict

# Go の場合
# go vet + staticcheck + errcheck
```

### Month 1: パイロット運用ルール

チームで最初に合意すべき3点です。

1. **AIに仕様を決めさせない**: Discovery・仕様・実装の順で人間の承認を挟む
2. **AIが生成したコードも同じゲートを通す**: PRレビュー・CI・テストは全件適用
3. **週次ふりかえり**: 「AIが便利だった点」「困った点」を記録し、プロセスを改善する

### Month 2-3: 効果測定

KPIを計測し、稟議書に書いた目標値と比較します。

目標を達成していれば、数字を持って「本格展開の提案」ができます。達成していなければ、「なぜ達成できなかったか」の分析を持って判断を仰ぎます。

どちらの場合も、データで語れる状態を作っておくことが重要です。

## まとめ: 明日の提案に使えるチェックリスト

提案書を出す前に確認してください。

**データで語れているか**
- [ ] 「AIが悪い」ではなく「品質基盤の不在が問題」という視点で語っている
- [ ] lint strictで防止できる件数（45%）を根拠として使っている

**ROIで語れているか**
- [ ] 投資額を時間×工数で計算している
- [ ] 回収見込みをバグ防止件数×修正コストで計算している
- [ ] 月次KPIを3〜4個定義している

**リスク対策で語れているか**
- [ ] 「AIが品質を下げる」懸念への回答がある（lint + ゲート設計）
- [ ] 「誰が責任を取るか」への回答がある（ゲート設計で人間が承認）
- [ ] 「効果がなかったらどうする」への回答がある（パイロット期間 + 判断基準の事前合意）

**運用プランがあるか**
- [ ] Month 0の品質基盤整備が具体的
- [ ] パイロット期間の運用ルールが決まっている
- [ ] 効果測定の方法と判断タイミングが明確

この3点（データ・ROI・リスク対策）を揃えた提案は、感想で語った提案とは別物になります。

「AIツール入れたいんですが」ではなく、「データとROIとリスク対策を持ってきました」と言える状態で、もう一度打席に立ってください。
