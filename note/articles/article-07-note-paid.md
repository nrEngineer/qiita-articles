## 原則1: Day 0に品質基盤を置く

### バグとの対応

46件の内訳を改めて整理します。

- print残留: 23件 --- `ruff T201` / `eslint no-console` / `forbidigo` で機械的に防止可能

### 判断フレームワーク: 品質基盤の投資回収期間

ここからが有料記事の本題です。「品質基盤を入れるべきか」ではなく、「いつ元が取れるか」を計算する方法をお伝えします。

**前提の数字**

私の経験から導いた目安です。プロジェクトの規模や言語によって前後しますが、桁が変わるほどのズレはないと考えています。

- 品質基盤の初期構築コスト: 約4時間（CI + lint + 型チェック + pre-commit）
- 1件のlint違反バグの修正コスト: 約30分（発見 + 原因調査 + 修正 + レビュー + 再デプロイ）
- 1件の型安全バグの修正コスト: 約1時間（発見 + 影響範囲調査 + 修正 + テスト + レビュー）

**計算方法**

```
投資回収件数 = 初期構築コスト / 1件あたりの修正コスト

lint系の場合:
  4時間 / 0.5時間 = 8件
  → 8件のlint違反バグを防いだ時点で元が取れる

型チェック系の場合:
  4時間 / 1時間 = 4件
  → 4件の型安全バグを防いだ時点で元が取れる
```

私のプロジェクトでは、lint系で35件、型安全で9件のバグが出ました。品質基盤があれば、これら44件全てを防止できました。

```
実際の節約効果:
  lint系: 35件 x 0.5時間 = 17.5時間
  型安全: 9件 x 1時間 = 9時間
  合計: 26.5時間の節約

初期投資: 4時間
投資回収率: 26.5 / 4 = 662%
```

4時間の投資で26.5時間のリターン。これが「Day 0に入れる」理由です。

**判断フレームワーク**

- もしプロジェクトの見込み期間が1週間未満なら → 最低限のlint設定だけ入れる（1時間）
- もしプロジェクトの見込み期間が1週間以上なら → lint + 型チェック + CI の3点セットを入れる（4時間）
- もしプロジェクトの見込み期間が1ヶ月以上なら → 上記 + pre-commit + カバレッジ閾値を入れる（6時間）
- もし既に数千行のコードがある状態なら → 新規ファイルだけstrict、既存はignoreリストで段階導入

### チームへの導入: 段階的アプローチ

「全部一気に入れる」はチームでは難しいです。反発が生まれます。

私が経験したのは「CIが赤になってマージできない！」という苦情でした。全員の開発フローが突然変わるのは、正しいことであっても摩擦を生みます。

段階的に導入する順序はこうです。

**Week 1: CIだけ入れる**

CIが存在するだけで価値があります。最初はlintもテストも入れず、ビルドだけ通す設定にします。「CIが回っている」状態を作ることが最初のステップです。

**Week 2: lintを追加する（警告のみ）**

CIにlintを追加しますが、最初は「警告」として出すだけにします。マージをブロックしません。チームメンバーに「こういう問題が検出されます」と見せることが目的です。

**Week 3: lintをエラーに変更する**

2週間で警告に慣れたら、エラーに切り替えます。ここからlint違反があるとマージできなくなります。この時点で既存コードの違反は`.ruff.toml`のignore設定で除外しておきます。

**Week 4: 型チェックを追加する**

lintが定着したら、型チェックを追加します。Pythonなら`mypy --strict`、TypeScriptなら`tsc --noEmit`です。

この順番のポイントは「最もコストが低く、効果が高いものから入れる」ことです。CIの導入コストは最も低く（ファイル1つ追加するだけ）、lintは最も多くのバグ（35件）を防ぎます。

## 原則2: Discoveryが先、コードが後

### バグとの対応

15件の仕様齟齬を再分析しました。

- 暗黙の前提条件が伝わっていなかった: 8件
- 具体的な条件が未定義だった: 4件
- エラー時の挙動が未定義だった: 3件

全てに共通するのは「具体例（Example）がなかった」ことです。

### Example Mappingの3つのレベル

無料記事ではExample Mappingの基本を紹介しました。ここでは、私が実際に使い分けている3つのレベルを紹介します。

**Level 1: Question List（5分）**

最もシンプルな形式です。機能名だけ書いて、「まだ決まっていないこと」をリストアップするだけです。

```
機能: お問い合わせ一覧

Questions:
- ページネーションは必要か？
- 1ページ何件？
- ソートは？
- 0件のときの表示は？
- 検索スコープは？
- 削除済みデータの扱いは？
```

これだけでも、「何も考えずにコードを書き始める」より圧倒的にマシです。Questionを書いてから実装を依頼すれば、AIは「ページネーションは必要ですか？」と聞いてくれる確率が上がります。

**Level 2: Rule + Example（30分）**

標準的なExample Mappingです。BDDの文脈で Matt Wynne が提唱した形式そのものです。

```
Story: 管理者として、お問い合わせ一覧を見たい

Rule 1: 一覧はページネーションされる
  Example A: 20件以下 → ページネーションなし
  Example B: 21件以上 → ページネーション表示
  Example C: 0件 → 「お問い合わせはありません」と表示

Rule 2: 各行をクリックすると詳細に遷移する
  Example A: ID "c-1" をクリック → /contacts/c-1 に遷移

Rule 3: ステータスでフィルタリングできる
  Example A: 「未対応」でフィルタ → 未対応のみ
  Example B: フィルタ解除 → 全件

Questions:
  - ソート機能は必要か？ → v1では不要
  - ページあたり件数は変更可能か？ → 20件固定
```

**Level 3: Given-When-Then + Test List（2時間）**

SFADの`/sfad spec`が出力する最も詳細な形式です。Level 2をさらに構造化し、テスト可能な形にします。

```
Story: 管理者として、お問い合わせ一覧を見たい

UC-1: ページネーション
  Given: 管理者がログイン済みで、お問い合わせが21件以上存在する
  When: お問い合わせ一覧ページにアクセスする
  Then: 20件が表示され、ページネーションが表示される
  AND: デフォルトは1ページ目が選択されている

  Given: お問い合わせが20件以下
  When: お問い合わせ一覧ページにアクセスする
  Then: 全件が表示され、ページネーションは表示されない

  Given: お問い合わせが0件
  When: お問い合わせ一覧ページにアクセスする
  Then: 「お問い合わせはありません」と表示される

Test List:
  - [ ] UC-1: 21件以上でページネーション表示
  - [ ] UC-1: 20件以下でページネーション非表示
  - [ ] UC-1: 0件で空メッセージ表示
  - [ ] UC-1: デフォルト1ページ目
  - [ ] UC-2: 行クリックで詳細遷移
  - [ ] UC-3: ステータスフィルタ（未対応）
  - [ ] UC-3: フィルタ解除で全件表示
  - [ ] S-1: ローディング中の表示
  - [ ] S-2: APIエラー時の表示
```

### 判断フレームワーク: どのレベルを使うか

- もし機能の複雑度が低く、不確実性も低いなら → Level 1（Question List）で十分
- もし機能の複雑度が中程度、または不確実性があるなら → Level 2（Rule + Example）を使う
- もし機能の複雑度が高く、複数のビジネスルールが絡むなら → Level 3（Given-When-Then + Test List）を使う
- もし「この機能、何をどう作るかまだよく分からない」と感じたなら → Level 3を使うべきサイン

**複雑度の目安**

- 低: CRUD操作、単純な一覧表示、静的ページ
- 中: 条件付き表示、フィルタリング、バリデーション
- 高: 状態遷移、権限制御、外部API連携、決済処理

**不確実性の目安**

- 低: 仕様が決まっている、類似機能の実績がある
- 中: 仕様が大まかに決まっている、エッジケースが不明
- 高: 仕様が曖昧、ステークホルダーの合意がない

### チームへの導入

「明日からExample Mappingをやりましょう」は失敗します。

最初は「Question Listだけ書く」から始めます。機能の実装を依頼する前に、「Questions」セクションだけ書く習慣をつけます。これなら5分で終わります。

Question Listを書く習慣が定着したら、Level 2に移行します。Questionに答えていくうちに、Ruleが自然と見えてきます。

## 原則3: テストは仕様の表現として書く

### Dan Northの名言とその意味

Dan Northは「テストは実装の検証ではなく、振る舞いの定義」と言いました。この言葉の実践的な意味を具体例で示します。

### 良いテスト名 vs 悪いテスト名

テスト名を読んだだけで「何の仕様か」がわかるかどうか。これが判断基準です。

**Python**

```python
# 悪い例: 実装の検証
def test_function_returns_true():
    assert is_valid(data) is True

# 良い例: 振る舞いの定義
def test_admin_can_view_all_contacts():
    admin = create_user(role="admin")
    contacts = get_contacts(user=admin)
    assert len(contacts) == total_count

# 悪い例: 何をテストしているか不明
def test_calculate():
    assert calculate(100, 0.1) == 110

# 良い例: ビジネスルールが読み取れる
def test_tax_calculation_rounds_down_to_two_decimal_places():
    assert calculate_tax(price=100, rate=0.1) == 110.00
```

**TypeScript**

```typescript
// 悪い例
it('works', () => {
  render(<ContactList />);
  expect(screen.getByText('お問い合わせ')).toBeInTheDocument();
});

// 良い例
it('displays error message when API returns 500', () => {
  mockApi.get('/contacts').reply(500);
  render(<ContactList />);
  expect(screen.getByText('データの取得に失敗しました')).toBeInTheDocument();
});

// 悪い例
it('should render', () => {
  const { container } = render(<Pagination items={items} />);
  expect(container).toBeTruthy();
});

// 良い例
it('hides pagination when total items are 20 or fewer', () => {
  const items = createItems(20);
  render(<Pagination items={items} />);
  expect(screen.queryByRole('navigation')).not.toBeInTheDocument();
});
```

**Go**

```go
// 悪い例
func TestLogin(t *testing.T) {
    err := Login("user@example.com", "password123")
    assert.NoError(t, err)
}

// 良い例
func TestLogin_LocksAccount_After5FailedAttempts(t *testing.T) {
    for i := 0; i < 5; i++ {
        Login("user@example.com", "wrong-password")
    }
    err := Login("user@example.com", "correct-password")
    assert.ErrorIs(t, err, ErrAccountLocked)
}

// 悪い例
func TestGetOrders(t *testing.T) {
    orders := GetOrders(userID)
    assert.NotNil(t, orders)
}

// 良い例
func TestGetOrders_ReturnsEmptySlice_WhenUserHasNoOrders(t *testing.T) {
    orders, err := GetOrders(ctx, newUserID)
    assert.NoError(t, err)
    assert.Empty(t, orders)
}
```

### 判断フレームワーク: テスト名の品質チェック

- もしテスト名を読んで「この機能がどう動くか」が想像できないなら → テスト名を書き直す
- もしテスト名に実装の詳細（関数名、変数名、returnの型）が含まれているなら → ユーザー視点の振る舞いに書き換える
- もしテスト名が「test_xxx」や「it('works')」のように汎用的なら → 「誰が」「何をしたとき」「どうなるか」の形式で書き直す

**テスト名のテンプレート**

```
Python: test_{主語}_{動作}_{条件}()
  例: test_admin_can_view_all_contacts()
  例: test_order_fails_when_stock_is_zero()

TypeScript: it('{条件}のとき、{結果}')
  例: it('displays error message when API returns 500')
  例: it('hides pagination when items are 20 or fewer')

Go: Test{主語}_{動作}_{条件}(t *testing.T)
  例: TestLogin_LocksAccount_After5FailedAttempts
  例: TestGetOrders_ReturnsEmptySlice_WhenUserHasNoOrders
```

## 原則4: Test Listを育てる

### Canon TDD (Kent Beck 2023) のTest List概念

Kent Beckが2023年に整理したCanon TDDには「Test List」というプラクティスがあります。実装を始める前に「テストすべきシナリオの一覧」を書き出しておくという考え方です。

ここで重要なのは「育てる」という部分です。最初から全部書き出せるとは限りません。実装を進める中で「あ、このケース考えていなかった」という発見が必ずあります。

### [ADDED]マークの運用

実装中に発見したシナリオには `[ADDED]` マークをつけてリストに追加します。

```
Test List: 注文処理

初期リスト（16項目）:
- [x] UC-1: 正常な注文が保存される
- [x] UC-1: 在庫が0のとき注文が失敗する
- [x] UC-1: 金額が0以下のとき注文が失敗する
- [x] UC-2: 注文一覧が取得できる
- [x] UC-2: ページネーション
- [x] UC-3: 注文詳細が取得できる
- [x] S-1: ローディング中の表示
- [x] S-2: エラー時の表示
- [x] S-3: 0件時の表示
- [x] S-4: 正常時の表示
... (他6項目)

実装中に追加（+3項目）:
- [x] UC-1: 在庫数が注文数ぴったりのとき注文が成功する [ADDED]
- [x] UC-2: 日付フォーマットが統一されて表示される [ADDED]
- [x] UC-3: 存在しないIDでアクセスすると404が返る [ADDED]

最終: 19/19 完了
```

### 判断フレームワーク: Test Listの使い方

- もし実装中に「あ、このケースどうなるんだろう」と思ったなら → その場でTest Listに [ADDED] として追加する。後回しにしない
- もしTest Listの完了率が50%を超えたら → 残りの項目を見直して優先順位をつける
- もしTest Listに20項目以上あるなら → 機能が大きすぎる可能性がある。分割を検討する
- もしTest Listの [ADDED] 項目が初期リストの30%以上あるなら → Discoveryが不十分だった。次回はLevel 3で仕様を書く

### 実例: 初期16項目から最終19項目まで

お問い合わせ一覧機能の開発で、私が実際に経験した流れです。

最初に16項目書き出しました。ページタイトル表示、データ表示、行クリック遷移、ページネーション、ステータスフィルタ、ローディング状態、エラー状態、空状態。

UC-2の実装中に「日付のフォーマットってどうするんだっけ」と気づきました。仕様に書いていなかったので、Test Listに [ADDED] として追加しました。

UC-4のページネーションを実装しているとき、「skip=-1を渡したらどうなる？」という疑問が浮かびました。これもTest Listに追加。

UC-3で存在しないIDへのアクセスを実装していて、「404のレスポンスフォーマットを決めていなかった」と気づきました。3つ目のADDED。

結果として16 + 3 = 19項目。この3項目は、Test Listを「育てる」習慣がなかったら見落としていたはずです。

## 原則5: Double-Loopで安全網を作る

### 外側と内側の役割

外側のループ（受け入れテスト）は「機能全体が動くか」を確認します。内側のループ（ユニットテスト）は「各部品が正しいか」を確認します。

```
外側ループ: 受け入れテスト = Red
  「お問い合わせ一覧が正しく表示され、操作できること」

  ↓ 機能がまだないので当然Red

内側ループ: UCテスト
  UC-1: ページタイトル表示 → Red → Green → Refactor
  UC-2: データ表示 → Red → Green → Refactor
  UC-3: 行クリック遷移 → Red → Green → Refactor
  UC-4: ページネーション → Red → Green → Refactor
  S-1〜S-4: 状態パターン → Red → Green → Refactor

  ↓ 全UCがGreenになった

外側ループ: 受け入れテスト = Green
  「機能全体が正しく動いている」
```

### 判断フレームワーク: 完了基準

- もし受け入れテストがGreenなら → その機能は「完成」と言い切れる
- もし受け入れテストがRedだが全UCテストがGreenなら → UCテスト間の統合に問題がある
- もしUCテストがGreenだが受け入れテストがRedなら → 受け入れテストのシナリオに漏れがある
- もし個別のUCテストがRedなら → その部品の実装がまだ完了していない

これが明確な完了基準です。「たぶん動いてると思います」ではなく、「受け入れテストがGreenです」と言えることが、Double-Loopの価値です。

### London School vs Chicago School の使い分け判断

無料記事で概要を紹介しましたが、ここでは「どの場面でどちらを使うか」の判断基準を詳しく書きます。

**London School（モック多め、振る舞い検証）を使う場面**

- もしテスト対象がAPIエンドポイントなら → Service層をモックして、入出力の契約を検証する
- もしテスト対象がUIコンポーネントなら → hooks/APIをモックして、表示の振る舞いを検証する
- もしテスト対象が外部サービスとの連携なら → 外部サービスをモックして、呼び出しパターンを検証する

**Chicago School（モック最小限、状態検証）を使う場面**

- もしテスト対象がビジネスロジック（サービス層）なら → 実際のオブジェクトを使い、処理結果の状態を検証する
- もしテスト対象がドメインモデルなら → 純粋な状態変換として、モックなしで検証する
- もしテスト対象が計算ロジックなら → 入力と出力だけで検証する

**レイヤー別の使い分け**

```
FE ページ/テンプレート  → London School（hooks/APIをモック）
FE コンポーネント       → London School（props/eventsを検証）
BE エンドポイント       → London School（Service層をモック）
BE サービス層          → Chicago School（状態検証）
BE ドメインモデル       → Chicago School（モック不要）
```

この使い分けをSFADの`SKILL.md`に「テスト流派の使い分け」として定義しました。Claude Codeがこのルールを読むことで、レイヤーに応じた適切なテストスタイルを選んでくれます。

## 原則6: 仕組みで防ぐ、人の規律に頼らない

### 「コードレビューで見つける」は再現性がない

人間のコードレビューに頼る防止策は、3つの理由で不安定です。

1. レビュアーの体調やスケジュールに依存する
2. 同じ人が同じ問題を毎回見つけるとは限らない
3. レビュアーが不在のとき、品質が下がる

「人が2回以上同じ指摘をしたら、それはlintルールにすべき」。これが私の判断基準です。

### 判断フレームワーク: 仕組み化の判断

- もしコードレビューで同じ指摘を2回以上したなら → lintルールまたはpre-commitフックで自動化する
- もしCI/CDパイプラインが存在しないなら → まずCIを作ることが最優先
- もしテストなしでマージ可能なら → ブランチ保護ルールを設定する
- もしフォーマットの揺れが議論になっているなら → フォーマッターを導入して議論自体をなくす

### 各言語のカスタムlintルールの作り方

既存のlintルールでカバーできない、プロジェクト固有のルールを追加したい場面があります。

**Python（ruff / flake8プラグイン）**

ruffは外部プラグインに対応していませんが、`per-file-ignores`と`select`の組み合わせで細かく制御できます。プロジェクト固有のルールが必要な場合は、`flake8`のカスタムプラグインを書きます。

```python
# flake8カスタムプラグインの骨格
import ast
from flake8.options.manager import OptionManager

class NoDirectDBQueryChecker:
    name = 'no-direct-db-query'
    version = '0.1.0'

    def __init__(self, tree: ast.AST):
        self.tree = tree

    def run(self):
        for node in ast.walk(self.tree):
            if isinstance(node, ast.Call):
                # db.query()の直接呼び出しを検出
                # Service層を経由すべき
                pass
```

**TypeScript（ESLintカスタムルール）**

```typescript
// eslint-rules/no-raw-fetch.ts
module.exports = {
  create(context) {
    return {
      CallExpression(node) {
        if (node.callee.name === 'fetch') {
          context.report({
            node,
            message: 'fetch()を直接使わないでください。apiClient を使ってください。',
          });
        }
      },
    };
  },
};
```

**Go（go vetカスタムアナライザー）**

```go
// cmd/customvet/main.go
package main

import (
    "golang.org/x/tools/go/analysis"
    "golang.org/x/tools/go/analysis/singlechecker"
)

var Analyzer = &analysis.Analyzer{
    Name: "norawsql",
    Doc:  "raw SQLクエリの使用を検出します",
    Run:  run,
}

func main() {
    singlechecker.Main(Analyzer)
}
```

カスタムルールを書くかどうかの判断基準は「コードレビューで3回以上指摘したか」です。2回なら偶然かもしれませんが、3回あれば構造的な問題です。

## 原則7: 複利を意識する

### 「とりあえず」の判断が6ヶ月後に223箇所の修正になった実例

無料記事でも触れた話ですが、ここではより詳しく経緯を書きます。

プロジェクト開始2日目。最初の関数を書くとき、「型ヒントは後で書こう」と判断しました。まだコード量が少なかったので、後からでも追加できると思ったのです。

1週間後。関数が30個になりました。型ヒントがない関数が30個。「今つけるのは面倒だから、もう少し後で」。

1ヶ月後。ファイル数が40を超えました。型ヒントがないコードが数百行。「mypyを入れたら大量のエラーが出るから、落ち着いてからにしよう」。

6ヶ月後。223箇所の型ヒント未設定。mypyを入れたら画面が真っ赤。修正に3日かかりました。

### 技術的負債の複利効果の計算

```
1日目: 型ヒントなし 1箇所
  修正コスト: 5分

1週間後: 30箇所
  修正コスト: 150分（2.5時間）

1ヶ月後: 約100箇所
  修正コスト: 500分（8.3時間）
  + 依存関係の調査: 2時間
  + テストの修正: 3時間
  合計: 約13時間

6ヶ月後: 223箇所
  修正コスト: 1115分（18.6時間）
  + 依存関係の調査: 8時間
  + テストの修正: 12時間
  + CIの調整: 4時間
  合計: 約42時間（≒ 5営業日）
```

1日目に5分かけていれば、6ヶ月後の5営業日がなくなっていました。これが技術的負債の複利効果です。

### 判断フレームワーク: 負債を取るべきか

- もしプロトタイプや検証目的のコードなら → 負債を取ってよい。ただし「捨てるコード」と明示する
- もしリリースするプロダクションコードなら → 負債を取るべきではない
- もし「3ヶ月後にこの判断はどう響くか」の答えが「影響なし」なら → 負債を取ってよい
- もし「3ヶ月後にこの判断はどう響くか」の答えが「修正箇所が増える」なら → 今修正する
- もしチームメンバーがこのコードを触る可能性があるなら → 負債を取るべきではない

**「この判断は3ヶ月後にどう響くか？」を毎回問う**

これが最もシンプルで効果的な判断基準です。私はこの問いをIDE のメモに貼っています。急いでいるときほど、この問いが効きます。

## 7原則の相互関係

7つの原則は独立しているのではなく、互いに補強し合います。

```
原則1（Day 0基盤）→ 原則6（仕組み化）→ 原則7（複利効果）
      ↓
原則2（Discovery）→ 原則3（テスト=仕様）→ 原則4（Test List）
                                              ↓
                                        原則5（Double-Loop）
```

**縦の流れ: 基盤を作り、仕組みにし、複利で効かせる**

原則1で品質基盤を置く。原則6でその基盤を人の規律ではなく仕組みとして定着させる。原則7でその仕組みが複利で効いてくる。

**横の流れ: Discoveryからテスト、実装へ**

原則2で仕様を具体化する。原則3でその仕様をテストとして表現する。原則4でテストリストを育てる。原則5でDouble-Loopの安全網で保護する。

この2つの流れが交差するところにSFADのコマンド体系があります。原則1は`/sfad init`で実現し、原則2-5は`/sfad cycle`で実現しています。

## 原則が矛盾するときの優先順位

### ケース1: 「Day 0基盤」と「とりあえず動かす」が矛盾する場合

プロトタイプを2日で作る必要がある。品質基盤を整えている時間がない。

判断基準: **リスクで判断する**

- もしプロトタイプが「捨てるコード」なら → 品質基盤を省略してよい。ただしコードのどこかに「PROTOTYPE: このコードは捨てる前提」と明記する
- もしプロトタイプが「動いたらそのまま本番に使われる可能性がある」なら → 品質基盤を入れる。最低限、lintだけでも入れる（30分）
- もしプロトタイプの後に本番コードを書く予定があるなら → プロトタイプは品質基盤なしでよいが、本番コードを書き始める前に必ず基盤を整える

### ケース2: 「テスト=仕様」と「スピード」が矛盾する場合

納期が迫っている。テストを書く時間がない。

判断基準: **変更頻度で判断する**

- もしこのコードが今後ほとんど変更されないなら → テストを後回しにしてもよい。ただし「テストなし」とTODOコメントを残す
- もしこのコードが今後頻繁に変更されるなら → テストを先に書く。変更のたびにデグレが起きるコストの方が大きい
- もしこのコードが「一度書いたら誰も触らない」ユーティリティなら → テストを省略してもリスクは低い

### ケース3: 「Discovery先行」と「顧客からのフィードバック待ち」が矛盾する場合

顧客が仕様を決められない。でもとりあえず動くものを見せてほしいと言われている。

判断基準: **不確実性の高い部分を分離する**

- もし全体の仕様が決まらないなら → 「決まっている部分」だけDiscoveryして実装する。決まっていない部分はQuestion Listに残して、モック表示にする
- もし「だいたいこんな感じ」という方向性はあるなら → Level 1のQuestion ListだけでOK。実装してフィードバックを得る
- もし完全に白紙なら → Discovery以前に、ペーパープロトタイプやワイヤーフレームで方向性を決めるべき。コードを書くのはまだ早い

## おわりに

7つの原則を並べると、結局のところ「当たり前のことを当たり前にやる」という話です。

でもその「当たり前」を78件のバグを踏んでからようやく言語化できた、というのが私の経験です。もっと早く体系的に整理していたら、78件のうち70件以上は防げていたはずです。

この記事で提供した判断フレームワークは、「もし〜なら〜する」の形式で統一しました。理由は、判断基準が曖昧なとき、人は楽な方に流れるからです。明確な判断基準があれば、「今回はどうする？」と迷う時間が減ります。

チーム導入ロードマップも、「一度に全部」ではなく「段階的に」という設計にしました。完璧を求めるよりも、最も効果が高いところから始める。それが78件のバグから学んだ最大の教訓です。
